                    <h2>Project Details</h2>
                    <p class="description-title">
                        For <a href="/project/1071">The HYDRA multiheaded virtual computer</a>
                    </p>
                    <p class="description-summary">The HYDRA is a computing platform using a small footprint Hypervisor to run 1-6 (or more) hardware assisted virtual machines of any x86 OS </p>
                    <div class="description-metainfo">
                        <span>created by
                            <a href="/hacker/2230-eric" title="Visit Profile">
                                eric
                            </a>
                            <span>&bull;</span>
                            a year ago
                        </span>
                    </div>

                

            </div>
        </div>

        <div class="container post-content">
            <p>The goal here is to make this hardware assisted virtualization platform as easy to use as it's bare metal counterpart.&nbsp; Or,&nbsp; the virtalization should be completely transparent to the end user. They should be "in the Matrix", in a virtual world unaware of the virtual nature of their constructed environment, yet endowed with "powers" that allow all kinds of amazing feats formerly thought impossible.&nbsp;</p><p>The Xbox One is completely "in the Matrix", with all management removed.&nbsp; Boots 2 VM's &amp; that's it, if they break it's back to Day1 snapshot for you.&nbsp; If you're going to be in the "Matrix" you want to be Neo....</p><p><img src="//cdn.hackaday.io/images/3535601399347970753.JPG"></p><p>But datacenter level management comes with the expectation that you at least have your head wrapped around the whole virtualization thing.&nbsp;&nbsp;</p><p>The HYDRA attempts to strike a balance. ESXi boots from the primary gpu, which gets "stolen" for passthrough during boot at which time the screen freezes. The VM assigned to this GPU boots and takes the screen.&nbsp; Starting other VM's or switching the OS on the primary display needs some interface......</p><p>The Hydra was originally developed 1.5 years ago to be a digital menu board system for a sub shop. Since then I have used this system as my primary PC (s) , and have 2 deployed in others use.&nbsp; This allowed me to get feedback and explore alternative, user friendly methods for the tasks a typical end user may encounter, as well as discover all the capabilities. </p><p>First try: Installed a vm with only 1 usb passed through for a wifi card, &amp; created a softAP. Wrote an Android app and got a cheapo Android tablet. This allowed power &amp; snapshot operations, even showed the VMware vga interface during boot over VNC, so if you were stuck in "startup repair" you would see it at the tablet. However, wireless deemed unreliable. </p><p>2nd try: ESXi has python, so I wrote a menu system that is accessed from the console of the server, or by ssh, or over serial. Created installable VIB patch to easily install the menu. This works very well, power, snapshot, even networking settings are available. But, the end user is required to use a text menu at the console.&nbsp; Should be ok, but as it turns out it's NOT.&nbsp;</p><p><img src="//cdn.hackaday.io/images/7738191399348160509.JPG"></p><p>I once accidentally assigned the primary GPU for passthrough on my system, the ESXi screen froze halfway booting.&nbsp; This was an HP Riloe II card, but it wouldn't pass through. I tried an NVidia PCI card and could actually assign it to a vm.&nbsp;</p><p>3rd try: built mgmt VM including all the tools, simple &amp; complex. Added the softAP. Set to autostart.&nbsp; Works very well, just wastes a GPU....</p><p>Redirecting the console to serial seems to be the answer.&nbsp; A microcontroller based serial console could handle all that is needed, as well as maybe perform other functions, such as HDMI KVM switching. </p><p>Current mgmt interface:&nbsp; I redirected TTY2 (techsupportmode) to com1. Now we have 6 GPU's for passthrough. Added bonus: modules loading are shown on the terminal during boot. The python menu will run on a laptop with putty, or even a palm3.&nbsp; I can't really require a laptop be attached for mgmt and there's only so many palm3's out there. Under development is a serial terminal menu device based on a ATMEGA328 with a 4x20 LCD.&nbsp; Very similar to: </p><p><a href="http://hackaday.com/2008/05/29/how-to-super-simple-serial-terminal/">http://hackaday.com/2008/05/29/how-to-super-simple-serial-terminal/</a></p><p>I am however using other libraries, to get the type of menu I want and support for a jog wheel rather than buttons.&nbsp; It will show 4 VM's with power status at a time, highlighting your selection, and have indicators for more above or below.&nbsp; Only power &amp; snapshot operations would be available, due to limited space. The unchangeable management VM would head the list, and allow full management capabilities.&nbsp; But for daily operations, the 4 line lcd will be an interface end users can deal with.&nbsp;</p><p>Like this:</p><p><img src="//cdn.hackaday.io/images/822191399348498047.jpg"></p><p>The HYDRA can be configured with 6 video cards, or with only one.&nbsp; Next prototype will be a mini-itx board with 2 GPUs.&nbsp; So a small dvd player sized box with the 4x20 lcd menu would by default boot for instance SteamOS, with maybe Win7 on the other GPU. From the menu, other OS's could booted instead, snapshots reverted to, or KVM to another GPU.&nbsp;&nbsp;</p><p>Other uses include:&nbsp;</p><p>Digital Menu board</p><p>Hotel business center</p><p>Library</p><p>Internet cafe</p><p>Office in a box</p><p>School computer lab</p>
