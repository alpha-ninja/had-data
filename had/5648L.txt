                            <h2><a href="/project/5648/log/19593-add-eight-motor-support">add eight motor support</a></h2>
                            <small class="authorship">2 months ago  &bull;
                                <a class="gray-link" href="/project/5648/log/19593#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-19593"><p>Through ebay I was able to purchase 30 additional motors from China. Before that point, I used to order the vibration motors from the following link: <a href="http://www.seeedstudio.com/depot/Mini-vibration-motor-20mm-p-2300.html"></a><a href="http://www.seeedstudio.com/depot/Mini-vibration-motor-20mm-p-2300.html">http://www.seeedstudio.com/depot/Mini-vibration-motor-20mm-p-2300.html</a>. The seeedstudio order is more reliable but a lot more expensive. By chance, the ebay motors seem to be from the same kind. There are some small differences with the DC resistance between <strong>each</strong> motor and also the resistance is changing after rotation. Maybe it's an indication for decreased commutator quality?!</p><p><img src="https://cdn.hackaday.io/images/1667941434435374300.jpg"></p><p><strong>Changing Hard- and Software at github</strong></p><p>I have tagged the four motor version in source code and in design at github. If anyone has already started to build the old version, you could still use the correct (older) versions:</p><p>TheSixthSense_XMega is tagged with V1.0</p><p>TheSixthSense-PCB is tagged with V1.1</p><p>I would prefer to use the newest version with eight motor support.</p><p><strong>Whats new</strong></p><p>There aren't so many (eight / 16 bit) PWM Pins on a XMegaE, because I2C data- and clock line is blocking two PWM pins. Thats no problem cause the pwm resolution doesn’t need to be very high. Its enough to have 32 steps to adjust the vibration force to your needs.  </p><p>I changed the hardware PWM to a software BAM (<a href="http://www.batsocks.co.uk/readme/art_bcm_3.htm"></a><a href="http://www.batsocks.co.uk/readme/art_bcm_3.htm">http://www.batsocks.co.uk/readme/art_bcm_3.htm</a>)</p><p>Here is my implementation to init a timer with 4 compare interrupts an overflow interrupt using the Atmel Software Framework:</p><pre class="hljs cpp"><span class="hljs-comment">#define TIMER_RESOLUTION 500 // 250 HZ at 2 MHz and div = 8</span>
//init BAM with one timer to supply <span class="hljs-number">8</span> motors with <span class="hljs-number">250</span> Hz
void init_bam(void)
{

/* Unmask clock <span class="hljs-keyword">for</span> ... */
tc45_<span class="hljs-built_in">enable</span>(&amp;TCC4);

/* Configure TC <span class="hljs-keyword">in</span> normal mode */
tc45_<span class="hljs-built_in">set</span>_wgm(&amp;TCC4, TC45_WG_NORMAL);

/* Configure period equal to resolution to obtain <span class="hljs-number">250</span>Hz */
tc45_write_period(&amp;TCC4, TIMER_RESOLUTION);

//no cc outputs
TCC4.CTRLE = <span class="hljs-number">0</span>;

//bam timing -&gt; <span class="hljs-number">5</span> Bit possible with ovf vector
tc45_write_cc_buffer(&amp;TCC4, TC45_CCA, TIMER_RESOLUTION &gt;&gt; <span class="hljs-number">4</span>);
tc45_write_cc_buffer(&amp;TCC4, TC45_CCB, TIMER_RESOLUTION &gt;&gt; <span class="hljs-number">3</span>);
tc45_write_cc_buffer(&amp;TCC4, TC45_CCC, TIMER_RESOLUTION &gt;&gt; <span class="hljs-number">2</span>);
tc45_write_cc_buffer(&amp;TCC4, TC45_CCD, TIMER_RESOLUTION &gt;&gt; <span class="hljs-number">1</span>);
tc45_<span class="hljs-built_in">set</span>_overflow_interrupt_callback(&amp;TCC4, ovf_interrupt_callback);

tc45_<span class="hljs-built_in">set</span>_cca_interrupt_callback(&amp;TCC4, cca_interrupt_callback);
tc45_<span class="hljs-built_in">set</span>_ccb_interrupt_callback(&amp;TCC4, ccb_interrupt_callback);
tc45_<span class="hljs-built_in">set</span>_ccc_interrupt_callback(&amp;TCC4, ccc_interrupt_callback);
tc45_<span class="hljs-built_in">set</span>_ccd_interrupt_callback(&amp;TCC4, ccd_interrupt_callback);

/*
* Enable TC interrupts
*/

tc45_<span class="hljs-built_in">set</span>_overflow_interrupt_level(&amp;TCC4, TC45_INT_LVL_LO);
tc45_<span class="hljs-built_in">set</span>_cca_interrupt_level(&amp;TCC4, TC45_INT_LVL_LO);
tc45_<span class="hljs-built_in">set</span>_ccb_interrupt_level(&amp;TCC4, TC45_INT_LVL_LO);
tc45_<span class="hljs-built_in">set</span>_ccc_interrupt_level(&amp;TCC4, TC45_INT_LVL_LO);
tc45_<span class="hljs-built_in">set</span>_ccd_interrupt_level(&amp;TCC4, TC45_INT_LVL_LO);
/*
* Run
*/
tc45_write_clock_<span class="hljs-built_in">source</span>(&amp;TCC4, TC45_CLKSEL_DIV8_gc);

}</pre><br><p>As you can see it’s rather easy to setup the timer with ASF support.</p><p>Before you could call this function you should define the callback functions for the interrupts. The source code for every interrupt callback is mostly the same. You have just to define which bit should be modulated:</p><pre class="hljs cpp"><span class="hljs-function"><span class="hljs-keyword">static</span> <span class="hljs-keyword">void</span> <span class="hljs-title">ovf_interrupt_callback</span><span class="hljs-params">(<span class="hljs-keyword">void</span>)</span>
</span>{
soft_bam_process(<span class="hljs-number">0x01</span>);
tc45_clear_overflow(&amp;TCC4);
}

<span class="hljs-function"><span class="hljs-keyword">static</span> <span class="hljs-keyword">void</span> <span class="hljs-title">cca_interrupt_callback</span><span class="hljs-params">(<span class="hljs-keyword">void</span>)</span>
</span>{
soft_bam_process(<span class="hljs-number">0x02</span>);
tc45_clear_cc_interrupt(&amp;TCC4, TC45_CCA);
}</pre><p>...and so one. </p><p>The function <em>soft_bam_process</em> turns on and off all the motors you could add and you could add so mutch motors you have free digital pins.</p><p>Lets take a look in this function:</p><pre class="hljs cpp"><span class="hljs-function"><span class="hljs-keyword">void</span> <span class="hljs-title">soft_bam_process</span><span class="hljs-params">(<span class="hljs-keyword">uint8_t</span> bitmask)</span>
</span>{
<span class="hljs-comment">//error if adding motors without changing source</span>
<span class="hljs-preprocessor">#<span class="hljs-keyword">if</span>(MOTOR_NUM != 8)</span>
<span class="hljs-preprocessor">#<span class="hljs-keyword">error</span> (add more motors here)</span>
<span class="hljs-preprocessor">#<span class="hljs-keyword">endif</span></span>

<span class="hljs-keyword">if</span>(motor_soft_bam[<span class="hljs-number">0</span>] &amp; bitmask) {
gpio_set_pin_high(M1_O);
} <span class="hljs-keyword">else</span> {
gpio_set_pin_low(M1_O);
}

<span class="hljs-keyword">if</span>(motor_soft_bam[<span class="hljs-number">1</span>] &amp; bitmask) {
gpio_set_pin_high(M2_O);
} <span class="hljs-keyword">else</span> {
gpio_set_pin_low(M2_O);
}</pre><p>Its good to have some precompiler errors if you try to add more motors and forgot to map this motors to there gpios. The <em>motor_soft_bam</em> array is the new compare register array you could change from the main loop. If you set one 8 bit value to zero the corresponding motor will not vibrate. A value of 31 turns on all bits and you will have the full DC voltage from the battery at the motor pins. The processing power to map the bits to their corresponding gpios is very light.</p><p><strong>Result</strong></p><p>In theorie accuracy of the cardinal direction could be maximum ± 22.5 ° with eight motors. But there are some additional errors: </p><ul><li>positioning of the motors at the anklet</li><li>sensitivity of the shin/skin</li><li>reading and filter errors from sensor or software</li></ul><p>Nevermind, I like the eight motor version. It is a lot more accurate than the four motors version. You could really <em>feel</em> the huge differences to the old version.</p><p><strong>TODOs</strong></p><p>It’s a good idea to make this anklet a bit smarter and more useful for people without a debugger. Of course the solution in this year is bluetooth low energy. TI has a new BLE Processor, this time with a Cortex M3 on the chip. I liked the old Sensortag from TI a lot and I have a hole bunch here but I never reprogrammed it because of the old 8051 microcontroller core (free compiler problem). Things changed and TI was releasing the CC2650 this days. </p><p>The new Sensortag has a 9D Sensor (MPU-9250) and an extension header… Enough to control the motors direct from the Sensortag CPU…</p><br>I will order a Sensortag and test this out...</div>

                            <div style="position:relative">
                                <div id="post-body-19804"><p>Before I switch over to a Bluetooth Low Energy solution I was ordering a new PCB (v1.3 see on github) with some added features:</p><ul><li>main power switch to fully disconnect the battery</li><li>250 mA fuse to make sure the anklet can’t be aflame</li><li>a small push button to start calibration process and future use</li><li>new “battery connector”/pin headers in position with Nokia 2600 battery terminals</li><li>micro match connector to disconnect the motors if needed</li></ul><p>I hope I will get the order from oshpark.com in three weeks (shipping time to europe). I will use my old stencil to get only the LSM303D on the board the other components could be easily added by soldering iron.</p><p>It’s interesting if the battery directly under the magnetometer IC is a problem. It should depend on the potential reservoir of ferromagnetic materials in the lithium battery.</p>Has anybody experience with that?</div>

                            <div style="position:relative">
                                <div id="post-body-22072"><p>I am afraid I am running a bit out of time for the quarter finals. :-/</p><p>I have received the PCB version 1.3 from OSH Park and cut down my old stencil in two parts to solder the most parts (especially the LGA-16 package) in my reflow oven. It looks complicated but it was working like a charm! The hinted pasting process:</p><p><img src="https://lh5.googleusercontent.com/D7KV981uT1SyFmLazhG4VTQrqPsMk79bS4tCRaLzNAasbP7JZhqAc5mo5CyVj4tnfMaZz1vhqPTZURXswr30PpliFWD9735IjcHCJpVs4KiygBEydPbCfOo-bnQ8TjA3hlsMU9A"></p><p>My mouser order was arriving right in time and I added the new push button, main power switch, fuse and micro-match connector.</p><p>I spend half an hour to try to fiddle the micro-match connectors together until I was realizing that I have ordered a micro-match ribbon connector for direct soldering. Unfortunate, but then again it was my first project with micro-match parts.</p><p>Obviously wrong for TheSixthSense project:</p><p><img src="http://www.wennmacher-electronic.de/files/cms/Micromatch%20Imagebild.jpg"><br></p><p>Correct (but must be a 16 pin type):</p><p><img src="http://www.conrad.de/medias/global/ce/7000_7999/7400/7470/7479/747905_LB_00_FB.EPS_1000.jpg"><br></p><p>I was also adding two cutted 90&#xB0; pin-headers for the battery connection. The battery fits well but must be hold with a zipties in place since I haven&#x2019;t designed a well fitting case. The new colorful ribbon cable with the vibration motors looks a bit like art.</p>I hope I will receive my new mouser order tomorrow and can start with the 2 minutes video for the contest and more importantly, I could test the new software functions, like starting the  calibration process with the new push button interaction.<p><img src="https://lh6.googleusercontent.com/Ez808-P0L1DBaJjQiGDn7PQBt6exKkJB61iCaLyJ2AiHgzey09yS673Hw1W6eGU6l1Vcx0Ukh8Dlj_5mlqt5QkgtC47b365XzZY-FmGQnx2LYep2vuPlOKpllPnYo_wIom4qlho"></p><p><img src="https://lh5.googleusercontent.com/j71gtOJ_77-9t5d9OazLHA9iVMc8Wwt2jxrwY0FKXLLemSZOGKZs2zscwPkNqOqdp8VCVSP0u8OnehD0etWCWvIb1YWvp9IAwoAg5IPPgcVF8YoKZy8G18-KUUBaBR5Jv9IAG0M"></p><p><img src="https://lh3.googleusercontent.com/7fpga0-3UqTwFKcBl9W3sCKGqYl0V_hLKoy9MWs2x574Lf5lowzAkc4KJerCWBBWRzrFyMoWWaCWENfYWqocRiRXkCDUW0kdJ3dGtK4-2mKU_5A13VlUUZCV7Z5fpE9XCQC0HgQ"></p></div>
