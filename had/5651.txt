                    <div class="description">This is a HDMI 2D GPU board for microcontrollers.                 <div class="section section-details">
                    <h5>
                        
                            Details
                        
                    </h5>
                    

                    
                        <div class="post-content details-content">
                            <p><p>One of the frustrating thing when you deal with small embedded systems is when you want to go for graphic capability (at least medium to high resolution) you must use Linux or another high level OS (and moreover not real time). The reason is the bloody closed source code driver for all these cheap ARM chip (Broadcom, TI, ...). There are some projects that try to do bare-metal on the chip (on RasPI for example) but that&apos;s time consuming.</p><p>Then I decided to launch HOMER. With HOMER, you&apos;ll have the key to do some beautiful graphics, GUI and even games on a classic PC monitor.</p><p>HOMER SPECIFICATION (TARGET):</p><p>- 1280x720 (HD) @ 8bpp @ 60Hz</p><p>- Full palette over 24bit</p><p>- Double buffering (2 SRAM blocs)</p><p>- 2D hardware acceleration (HOMER command example: line(0,0,120,500,option) )</p><p>- FPGA config file USB upgradable (nobody wants to buy a blaster II)</p><p>- ESD protected</p><p>- HDMI output, audio included</p><h1>How Homer works (seethe sketch) ?</h1><p><img src="https://cdn.hackaday.io/images/99801436212066793.jpg"></p><p>Homer can connect toany host with a SPI interface. Supplying Homer is done via SPIconnector, via USB interface, or both.</p><p>The host (uC,Arduino, ESP8266, whatever...) sends high level commands to the SPIinterface (the max speed is not yet defined, I&apos;m using 10Mbps now).The commands are then pushed to the FIFO. The 2D GPU engine pull thecommands and process them. Basically, the engine contains a set ofprimitives (drawLine, putTxt, circle, fill, blobCopy,selectPalette,...) that read/write the non-displayed frame buffer(FB). Some primitives like putTxt and blobCopy take data from theFLASH and send them to the non-displayed FB. The flash contains userpre-loaded fonts, glyphs and images that are transferred (withprocessing) to FB with single commands. The host has only to storethe useful data in FLASH at once. A special command (SwapBuffers() orwhatever the name) ask HOMER to switch the FB that is displayed.Internally, the 2D engine ask the FB controller to do the switch. Assoon as the FB controller did it, it sends an ack to the 2D GPUengine, the following commands are then applied to the newnon-displayed FB. Transparency management is foreseen a&#x336;l&#x336;t&#x336;h&#x336;o&#x336;u&#x336;g&#x336;h&#x336; &#x336;i&#x336;t&#x336; &#x336;i&#x336;s&#x336; &#x336;s&#x336;l&#x336;o&#x336;w&#x336;e&#x336;r&#x336; &#x336;(&#x336;a&#x336; &#x336;&#x201C;&#x336;r&#x336;e&#x336;a&#x336;d&#x336; &#x336;b&#x336;e&#x336;f&#x336;o&#x336;r&#x336;e&#x336; &#x336;w&#x336;r&#x336;i&#x336;t&#x336;e&#x336;&#x201D;&#x336; &#x336;o&#x336;f&#x336; &#x336;F&#x336;B&#x336; &#x336;i&#x336;s&#x336; &#x336;n&#x336;e&#x336;c&#x336;e&#x336;s&#x336;s&#x336;a&#x336;r&#x336;y&#x336;)&#x336;. Displaying thecontent to the screen is done via a simple FB scanning: the CPLD hasa counter, synchronised with the FPGA one, that increment the FB RAMaddresses. The RAM data are transferred to FB controller then to oneof the user-stored palettes. Actually, palettes are LUT with 256entries and 24bits outputs (8bit per color). The data are thentransferred to the HDMI PHY chip with the right timing.</p><p>Typically, a hostprogram looks like this:</p><pre class="hljs cpp">selectPaletteStrings(<span class="hljs-number">3</span>);selectPaletteBlobs(<span class="hljs-number">1</span>);Loop() {Clear(); <span class="hljs-comment">/// or fill(...) or nothing</span>putString(X,Y, fontIDX, &#x201D;blablabla&#x201D;, foregroundColor, backgroundColor);...copyBlob(flashMemAddress, X, Y, W, H, operation); <span class="hljs-comment">///operation = bitwise</span>...drawLine(X1, Y1, X2, Y2, color, option); <span class="hljs-comment">///option will include antialiasing (slower)</span>...swapBuffers();}</pre><p>Host can implementthe commands like this (big endian, SPI 8 bits):</p><p>Example fordrawLine(X1, Y1, X2, Y2, color, option):</p><pre class="hljs cpp">buff[<span class="hljs-number">0</span>] = X1U; buff[<span class="hljs-number">1</span>] = X1L; buff[<span class="hljs-number">2</span>] = Y1U; buff[<span class="hljs-number">3</span>] = Y1L;buff[<span class="hljs-number">4</span>] = X2U; buff[<span class="hljs-number">5</span>] = X2L; buff[<span class="hljs-number">6</span>] = Y2U; buff[<span class="hljs-number">7</span>] = Y2L;buff[<span class="hljs-number">8</span>] = color; buff[<span class="hljs-number">9</span>] = option;sendSPI(buff, <span class="hljs-number">10</span>); <span class="hljs-comment">/// send the buffer to SPI, length = 10</span></pre><p>The Frames PerSecond (FPS) is 60 maximum, it corresponds to the HD refresh rate of60Hz. Depending of the 2D engine charge, the FPS may be lower.</p>
                                
                            </p>
                        </div>
