                            <h2><a href="/project/6503/log/20214-what-ive-got-so-far-and-what-i-need-to-figure-out">What I&#39;ve got so far, and what I need to figure out</a></h2>
                            <small class="authorship">2 months ago  &bull;
                                <a class="gray-link" href="/project/6503/log/20214#discussion-list">
                                    2 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-20214"><p>What I've got so far:</p><ul><li>Array of 64 speakers, divided into groups of four. Each group of four is driven by a TI LM48901 spatial audio beamforming &amp; amplifier chip, which does beamforming with its four speakers, while a central Teensy 3.1 does beamforming with the 16 groups.<br><img src="https://cdn.hackaday.io/images/14591435620269142.png"></li><li>Audio is sent from the Teensy to the LM48901s via I2S. To achieve the phase delays for beamforming, one of the following methods could be used:<ul><li>Boomers with integrated wavetables, so they can do their own phase delays (I don't think these exist; the LM48901 definitely doesn't have this feature.)</li><li>Very fast multiplexing of I2S, either with dedicated I2S multiplexers or clock-switching multiplexing</li><li>Programmable I2S delay lines (These could be made out of microcontrollers or FGPAs + memory or maybe shift registers.)</li><li>DAC + programmable analog delay lines (Suboptimal, and I don't know if those even exist.)</li><li>Free-running SDRAM(s) fed with I2S wavetable(s), feeding out I2S signals from different locations in memory for the different channels' phase delays (Do you think this would work, <a data-mention-user-id="40107" href="/hacker/40107">@esot.eric.wazhung</a>?)</li><li>The same thing but with a multi-readout-location ROM of some kind (I don't know if that exists.)</li></ul></li><li>Because the array will be emitting only a single frequency and the steering will be static or at least relatively slowly changing, the Teensy's beamforming task shouldn't be too difficult. Just a bit of trigonometry… (famous last words!)</li><li>The speakers will probably be set quite far apart on the ground to get the necessary separation for effective beamforming—tens to hundreds of meters probably. This will necessitate accurate measurement of each speaker's location, which can be done using a handheld GPS unit, and some kind of robust long-distance communication and synchronization scheme. (Maybe give each one its own GPS receiver so they can use GPS time, but that would make it very complex and expensive.)</li></ul><p>What I need to figure out:</p><ul><li>What frequency of sound is best for making water droplets in the cloud coalesce? <a href="https://www.reddit.com/r/AskScienceDiscussion/comments/3bkb6z/what_is_the_acoustic_resonant_frequency_of_water/" target="_blank">/r/AskScienceDiscussion thread</a></li><li>Which of the above methods for getting sound from the Teensy to the LM48901s is best?</li><li>What size and shape would work best for the speaker array?</li><li>How much power is required?</li><li>How can I generate LM48901 beamforming coefficients on the Teensy to accommodate array reconfiguration? (TI has a <a href="http://webench.ti.com/sawa/#main" target="_blank">web-based configuration generator</a>, but it (as well as the LM48901) is no longer supported, and it's intended for small-scale applications. Therefore I think I will have to reverse-engineer its output (or just read the datasheet, because I think it explains the coefficients) and implement a coefficient generator on the Teensy.)</li><li>and a whole lot more…</li></ul></div>
