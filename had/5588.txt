                    <h5>Description</h5>
                    <div class="description">NOTE: There is a new version of this project. Please read Project Log #3 (Hacker's True RNG). New schematics diagram is added to the existing one.<br /><br />This is a hardware password manager, built with PIC18F2455. Computer sees it as a standard USB HID keyboard, so it works on every computer and needs no extra software. When you are asked to type the password, you just plug it in USB port and press the corresponding key.<br /><br />So...<br />...no manual password typing...<br />...no driver installation...<br />...no password keeping in the computer...<br />...no crackable PRNG password creation...<br />...no peeking over your shoulder while you type the password...<br /><br />Dimensions are 76x24x12 mm, which is not bigger than most USB sticks. It uses the internal white noise TRNG to generate your passwords, which are held in internal MCU flash.<br /><br />This is an open project, and the source code (in MPLAB assembler) is available at www.voja.rs/PROJECTS/npr.htm.<br /><br />[Illustration by Bob Zivkovic]</div>

                <div class="section section-details">
                    <h5>
                        
                            Details
                        
                    </h5>
                    

                    
                        <div class="post-content details-content">
                            <p><p>Some more details...</p><p>All passwords are 15...20 character in length (which is random).</p><p>At least one lowercase, one uppercase character and one digit is guaranteed in each password</p><p>There are no punctuation marks or special characters, as not all servers accept them. But I shall add that option in setup, so you can choose it.</p><p>USB stack is open-source assembler software, by Bradley A. Minch. I have modified it so that it accepts uppercase/lowercase charracters, servicing &lt;shift&gt; bit in USB packet.<br></p><p>My friend asked me: Why don&apos;t you embed some PIN code, which can be typed directly on the unit, so that nobody can &quot;borrow&quot; it form you and use it? Good point, but you don&apos;t have to type PIN on this small and uncomfortable keyboard, you can add it at the more convenient way: type some extension on the existing passwords. So, when you generate new password, use the PC keyboard to type your &quot;pin&quot; on after it, and then store the extended password (there will be no conflict between USB keyboards on the computer). You have to do the same each time you use your password. This can be the same sequence of (say 4, or so) characters for all passwords.</p><p>GitHub link: <a href="https://github.com/vantonic/password_manager">https://github.com/vantonic/password_manager</a></p><p>Here&apos;s a brief demo:</p><p><a target="_blank" href="http://youtu.be/sQJSBvxbeYM"></a><iframe width="420" height="315" src="https://www.youtube.com/embed/sQJSBvxbeYM" frameborder="0" allowfullscreen=""></iframe><br></p>
                                
                            </p>

                            <h2><a href="/project/5588/log/23341-version-b-prototype">Version B prototype</a></h2>

                            <p><p>My PCBs arrived at last, so I can build the final prototype, with much simpler hardware.</p><p>The main modification is in True RNG hardware - istead of external white noise generator, which also worked fine, now I use the uninitialized internal RAM state. I described that process at <a href="http://hackaday.com/2015/06/29/true-random-number-generator-for-a-true-hacker/">http://hackaday.com/2015/06/29/true-random-number-generator-for-a-true-hacker/</a>, so I won&apos;t write about it here.</p><p>I have built six units (that much I promissed to my friends), and everything works great. Here is the first photo:</p><p><img src="https://cdn.hackaday.io/images/7635101439822138200.jpg"></p><br>

                        <li>
                            <h2><a href="/project/5588/log/19127-hackers-true-rng">Hacker&#39;s True RNG</a></h2>
                            <small class="authorship">2 months ago  &bull;
                            <a class="gray-link" href="/project/5588/log/19127#discussion-list">
                                2 comments
                            </a>
                            </small>
                            <p><p>How to save a lot of components, time and board space? White noise TRNG (True Random Number Generator) works fine - it is the standard solution for low-cost (and yet relatively high quality) random numbers, but it needs a lot of external components and takes two thirds of PCB space. Why can't we use some software PRNG (Pseudo RNG) instead of it?</p><p>Most gaming machines use that solution, and it works fine, but in this case it won't be a good idea. Here's why.</p><p>PRNG starts from some <em>known</em> seed and it always produces the same sequence when initialized with the same state. That's why gaming machines constantly employ their PRNGs to generate dummy random numbers (and thus new seeds) frequently. When someone starts playing the game, he can't say how many times seed was changed, so it appears as truly random number generator.</p><p>This project works in the other way - if you want to create the new password, you plug the unit in USB port and press two buttons (new-A...D). It could be very fast and thus predictable, even if you have a good PRNG with long sequence. But if we could generate the <em>truly </em>random seed for PRNG, we could solve the problem.</p><p>PIC MCUs don't have internal hardware for TRNG, but there is one resource which we can use, specifically with this project. When you apply power to the MCU, its internal Data RAM will have the unknown contents. Each flip-flop will be initialized to 0 or 1 state, which could the consequence of imperfection of internal circuits, but in most cases it will depend on thermal noise. That's why the contents of RAM is different each time it's switched on.</p><p>This feature is used for seed generation. Some flip-flops will always have the same state, but it doesn't matter, as most of them will be random. Our TRNG routine has to generate only eight bytes of TRNG sequence, so we can use many bits to make only one random bit. It will XOR the first 256 bytes of RAM to generate first byte, and so on - 256 bytes guarantee that there will be enough truly random (or, to be more specific, unpredictable) bytes. As this MCU has 2K of Data RAM, we shall have eight truly random 8-bit numbers. Four of them are used as 32-bit PRNG seed, and the remaining four are used for additional scrambling of PRNG results: if bit 0 in byte 0 (or 1,2,3) is set, then the result byte will be XORed with byte 4 (or 5,6,7), which was generated during initial RAM XORing. There is one more scrambling at the end - randomization sequence is constantly dummy invoked at about 70 KHz rate while MCU is waiting for the key to be pressed.</p><p>All this guarantees that excluding of TRNG hardware (white noise generator and DC-DC convertor) won't cause any performance degradation. The same procedure of seed creation can be used for other applications, keeping in mind the main thing - RAM contents <em>must</em> be uninitialized and used as-is, immidiatelly after power on. Here we need chaotic (high entropy) RAM contents, so we can't use no battery backup or high capacity decoupling - keep in mind that CMOS RAM retention voltage can be pretty low. That is why there is no RAM initializaton in firmware (except the small "housekeeping" portion), and there is 1K resistor in parallel with MCU supply line (bottom left on the yellow highlighted schematics diagram), it guaranees that RAM will not remember anything from its previous "reincarnations"!</p><p><img src="https://cdn.hackaday.io/images/5997021433837261958.jpg"></p><p>There is one more suggestion. Instead of 18F2455 (or 2550), it is better to use 18F24K50 (or 25K50). It is significanly cheaper (around half a price), it has internal clock source accurate enough for USB application, and internal MCLR pull-up.</p><p>Now there are only a few electronic components. USB and ICSP connectors, slightly smaller keys, LEDs and MCU (now in SSOP package) are still here, but there are only two capacitors and three resistors - that's all. I made the new PCB project, here it is compared to the first version. Dimensions of the final unit (excluding USB connector) will be 65x17x9.5 mm.</p><p><img src="https://cdn.hackaday.io/images/6582761433835377935.jpg"></p><p>And the component placement:</p><p><img src="https://cdn.hackaday.io/images/1156541435756126292.jpg"></p><p>Here are some tests of...</p>
