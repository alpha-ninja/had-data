                            <a class="gray-link" href="/project/5935/log/19665#discussion-list">
                                0 comments
                            </a>
                            </small>
                            <p><p>Ok, it's been some time since last log, and I changed a lot of things. Until then, I worked a lot on the hardware, and just coded a basic pygame script to experiment my changes. So, this code was very unoptimized, with one single big loop. It was time to work on this part.</p><ul><li>I started to rewrite the code to get more performance from it. I threaded the 3 main parts : usb capture, image processing, and displaying. From there, I could measure "framerates" of those threads, and find the bottlenecks.</li></ul><ul><li>First obvious bottleneck was usb capture framerate, locked around 10fps. The USB is rather limited on the Raspberry Pi, especially on B : first, USB and ethernet share the same chip, so bandwith suffers. Second, the RPi powering circuit is very basic, and USB could only get ~200mA. So I decided to hack the board and directly route the 5V from power input to USB. This did the trick, and I could capture at almost full speed (27-30fps). I documented this hack  <a href="http://hardware-libre.fr/2015/06/r-pi-1-corriger-le-probleme-dalimentation-des-ports-usb/">here</a>. </li></ul><table><tbody><tr><td><img src="https://cdn.hackaday.io/images/7511411434590763933.jpg"></td><td><img src="https://cdn.hackaday.io/images/1029611434590774340.jpg"></td></tr></tbody></table><ul><li>Second bottleneck is in the display thread. I found I reached the SPI max bandwidth : if I remove 1 screen, framerate jumps to 22-24fps. So, after many tweaks, I could not get more than 10-12 fps on each screen. This is not much, but it's very stable and both displays are perfectly synced. Unfortunately, I don't find a solution to this, except using a HDMI display I don't have (yet). </li></ul><ul><li>At this point, I decided to try replacing the USB camera with the Pi camera module (NoIR version) to get a better image. So I built an additional shell with cardboard to hold the module, while allowing to slide in and out. It also hides part of the cables, and I painted it black. </li></ul><p><img src="https://cdn.hackaday.io/images/5183771434590197625.jpg"></p><ul><li>Then, I had to find a way to capture Pi camera module at full speed in pygame. Adafruit uses a nice trick in their camera script, but 10fps was not enough. So I ended using an <a href="http://www.linux-projects.org/modules/sections/index.php?op=viewarticle&amp;artid=14">alternative UV4L driver</a>. With it, the Pi camera is used as a regular webcam, but it still uses the GPU. So, my pygame script could capture at full speed. It is running so nice that I decided to write a python lib for it.</li></ul><ul><li>I added a GY-80 10 axis IMU to the device. This is a gyro/mag/accel/baro combo I'm using in a custom headtracker I built to play Elite: Dangerous and flight sims (working very well by the way, I plan to sell it on Tindie as soon as I rebuilt it wireless). This IMU is a bit tricky to use, and I had to partially write 5 python libs to make it work. It is still not very well calibrated, especially the magnetometer. But it's ok for now.</li></ul><p><img src="https://cdn.hackaday.io/images/5338841434590131094.jpg"></p><ul><li>With the IMU working, I added some "telemetry" to the display : 2 lateral scales show pitch ant altitude, and a central bar gives the horizon. I don't use roll and yaw for the moment. I also show the ambient temperature in a corner.</li></ul><ul><li>Finally, I added a single button to the breakout board. The python code I wrote can do few things with this single button : short press calls/stops the HUD script, a longer press reboots the device, and an even longer one power it off. From now, I don't have to plug a keyboard or access it remotely anymore, except when developing on it.</li></ul><p><img src="https://cdn.hackaday.io/images/544831434590222825.jpg"></p><p>So, so far, the device is running well. The biggest problem is about the displays. Once I will have upgraded to a HDMI display, I won't be limited by framerate. I also plan to upgrade to a Raspberry Pi 2, to give some room for OpenCV video processing. I have so many ideas at this point...</p><p>But even at 10fps and with 240*320 resolution per eye, it's a great device and I don't feel any VR sickness with it.</p><p>I will release the full code on my GitHub once the cleanup is finished.<br></p><p>A last detail : I tried to use a BeagleBone at a point, because I have a very nice 7" 1024*600 display for it. But I quickly found the USB bandwith is also limited, and I could not capture at more than 10fps with a USB webcam. There's nothing I could do to it,  so I abandonned the idea. I don't want to kill my only beagleboard doing a similar hack I did on the Pi. <img src="https://cdn.hackaday.io/images/5180261434590335634.jpg"></p>
                                
                            </p>
                        </li>
                    
                        <li>
                            <h2><a href="/project/5935/log/19146-mkii-first-hardware-modifications">MkII : first hardware modifications</a></h2>
                            <small class="authorship">2 months ago  &bull;
                            <a class="gray-link" href="/project/5935/log/19146#discussion-list">
                                0 comments
                            </a>
                            </small>
                            <p><p>The first version of the HMD had a few little problems :</p><ul><li>one screen flickers a little because of the connections</li><li>PS3 eye camera capture quality is not so great</li><li>elements on every face is a problem to put the device on a table</li><li>some light is entering the device, from the borders of the screen.</li></ul><p>So I decided to make some modifications :</p><ul><li>I added my DIY arduino hat (from my domotic project) because connections are good, there's a I2C level shifter that will be usefull, and both SPI channels are fully exposed. There are also an ATmega328 running at 3.3v/8MHz, and a Microchip MCP1702 is providing 3.3V to the board from the Raspberry Pi 5V. I think I will build a new board without the atmega, but with added features for the HMD (IMU, RGB and IR leds, on/off button, for instance).</li><li>I replaced the PS3 camera with a Xbox360 camera (with IR filter and case removed)</li><li>I created a cardboard "shell" to put on top of the screens. It's painted black on the inside, and I added a fixture for the camera module.</li></ul><p>All these modifications allow me to put the raspberry on the top of the HMD. It's far less bulky, more robust and I can put it on a table...<br></p><p>With these modifications, I can say the issues I was facing are totally gone : image quality is far better, no more light is entering the device, and flickering has stopped.</p><p>Here are a few pictures, see more on <a target="_blank" href="http://wp.me/p4rFsK-tp">this page</a>.</p><p><img src="https://cdn.hackaday.io/images/1077681433719536471.jpg"><img src="https://cdn.hackaday.io/images/5548881433719551518.jpg"><img src="https://cdn.hackaday.io/images/9907811433719648559.jpg"><img src="https://cdn.hackaday.io/images/9276971433719569346.jpg"></p><p><img src="https://cdn.hackaday.io/images/5675081433719613773.jpg"></p><p>Better look, isn't it ?<br></p>
                                
                            </p>
                        </li>
                    
                        <li>
                            <h2><a href="/project/5935/log/18615-day-2-mark-i-assembling">Day 2, Mark I : assembling</a></h2>
                            <small class="authorship">3 months ago  &bull;
                            <a class="gray-link" href="/project/5935/log/18615#discussion-list">
                                0 comments
                            </a>
                            </small>
                            <p><p><a target="_blank" href="http://hardware-libre.fr/2015/05/openvar-le-casque-vr-do-it-yourself-mark-i-etape-2/"><em>Full link in French</em></a><br></p><p>In Day 1, I made both SPI displays to work in clone mode. They <em>could</em> achieve 50fps which is very good, and they seem to be synced (I will have to check this when the helmet is working)</p><p>Today, I'm starting to physically build the device.</p><p>Here is how both displays are connected :</p><p>Common to both displays :</p><ul><li>MOSI (pin 19)</li><li>MISO (pin 21)</li><li>SCLK (pin 23)</li><li>3.3v</li><li>GND</li></ul><p>For each display :</p><ul><li>Raspberry SP0 and SP1, which are SPI cable select (pins 24 and 26),  will go to SP0 on each display (pin 24),</li><li>GPIO 24 (pin 18) and GPIO 25 (pin 22) will go to each display's pin 22.</li></ul><p>Because I want to reuse displays in other projects in the future, I don't want to alter them. So I will build some adapters, to go from both display's 26 pins connector to the Raspberry Pi's 26 pins connector.</p><table><tbody><tr><td><img src="http://hardware-libre.fr/wp-content/uploads/2015/05/20150521_145540-300x225.jpg"></td><td><img src="http://hardware-libre.fr/wp-content/uploads/2015/05/20150521_152132-300x225.jpg"></td></tr><tr><td><img src="http://hardware-libre.fr/wp-content/uploads/2015/05/20150521_152152-300x225.jpg"></td><td><img src="http://hardware-libre.fr/wp-content/uploads/2015/05/20150521_161027-300x225.jpg"></td></tr></tbody></table><p>Unfortunately, after a quick test in the shell, I notice I can't use the displays horizontally : they are too far away from each other to fit the lenses. So, I will use them vertically.</p><p><img src="http://hardware-libre.fr/wp-content/uploads/2015/05/20150521_172928-300x225.jpg"></p><p>When used vertically, both display fit perfectly in the ColorCross shell.</p><p>I can now test the lenses alignment. Once perfect alignment is found, I will attache both displays to the shell with a little hot glue (sticks well and is easy to remove)</p><table><tbody><tr><td><img src="http://hardware-libre.fr/wp-content/uploads/2015/05/20150521_173031-300x225.jpg"></td><td><img src="http://hardware-libre.fr/wp-content/uploads/2015/05/20150521_173125-300x225.jpg"></td></tr></tbody></table><p><em>Displays are rotated in the pictures, but the overlay I linked to in Day 1 corrects this.</em></p><p>Without any surprise, view is pixellated, but it's not really important right now, and I have plans to drastically improve it in a future upgrade.</p><p>Good news is : displays are perfectly synced ! So this build is viable, after all !<br></p><p><br><em></em></p><table><tbody><tr><td><img src="http://hardware-libre.fr/wp-content/uploads/2015/05/20150521_190021-300x225.jpg"></td><td><img src="http://hardware-libre.fr/wp-content/uploads/2015/05/20150521_190027-300x225.jpg"></td></tr></tbody></table><p>PS3 Eye camera and Raspberry Pi have been temporary attached on top and bottom of the shell with velcro. </p><p>Very sexy, isn't it ? :D <br></p>
                                
                            </p>
                        </li>
