                <div class="section section-description">
                    <h5>Description</h5>
                    <div class="description">The alarm that came with my car used a standard encoding chip MC145028 with fixed code and modulated at 433Mhz. This is pretty simple but also very unsafe.<br />I was able to capture the RF signal using a SDR dongle and easily decoded the remote's code.<br />This is not security in my point of view.<br />So I planned on using two nRF24LE1 modules, one on the remote and one on the car, connected to the alarm. <br />They use the 2.4Ghz band that might be a bit crowded and doesn't allow for a big range but in this case I don't pretend to have a huge range. <br />The announced 100 meters in LOS is more than enough. This also prevents from unwanted activation/deactivation from accidental button presses on the remote.<br />This way I can implement rolling code feature and also AES encryption (already in a library on the SDK) in a very easy manner. <br /></div>
                </div>
            

            
                <div class="section section-details">
                    <h5>
                        
                            Details
                        
                    </h5>
                    

                    
                        <div class="post-content details-content">
                            <p><p>I bought two modules from eBay, one for the remote control as transmitter and one as the receiver in the car, connected to the alarm.</p><p>To program them I had to find a programmer. As they are a bit pricey I thought of building one myself. But It was getting a bit of a pain and meanwhile  I found a guy on the net that was using the Raspberry Pi to program them. I got mine out of the box and got installing the necessary stuff.</p><p>Then I connected a module to it (will update with schematic). It uses an SPI interface to download the binary code to the module. Got an example code compiled, downloaded and voila!!! </p><p>Got a led blinking! The hardware "Hello World" :D</p><p>Now for some real fun stuff.</p>
                                
                            </p>

                            <p><p>My estimates (by thumb rule) where a bit pessimist.</p><p>So, I got an average of 6mA standby current on the receiver module and only 2.6mA active current on the transmitter! This enables a much longer battery life!</p><p>The receiver standby current:</p><p><img src="https://cdn.hackaday.io/images/7800861436346454458.jpg"></p><p>The transmitter active current:</p><p><img src="https://cdn.hackaday.io/images/9192771436346413481.jpg"></p><p>P.S. don't mind the dusty multimeter :) It's been a while since he was last used.</p>

                        <li>
                            <h2><a href="/project/5897/log/20292-all-about-power">All about power...</a></h2>
                            <small class="authorship">2 months ago  &bull;
                            <a class="gray-link" href="/project/5897/log/20292#discussion-list">
                                0 comments
                            </a>
                            </small>
                            <p><p>Running the modules at full speed (16MHz) and adding the current consumption of the radio blocks on the TX and RX modules I noticed that the receiving will cause a lot of battery drain:</p><p>RX: ~13mA (radio) + ~9mA CPU = ~22mA</p><p>TX: ~11mA (radio) + ~9mA CPU = ~20mA</p><p>So I've dropped the CPU frequency to 1MHz and "PWMed" the radio block disabling it for a period of time (around 150ms) and enabling it for a short period (~50ms). This will enable a power saving around 75% on the radio alone. The CPU will drop to around 1mA making the total current of :</p><p>RX: ~4mA (radio) + ~1mA CPU = ~5mA<span class="redactor-invisible-space"><br></span></p><p><span class="redactor-invisible-space">This is more acceptable. </span></p><p><span class="redactor-invisible-space">The 12V rail current drain will be even lower as the switch mode power supply will draw less than that from the 12V.</span></p><p><span class="redactor-invisible-space">I'm going to make some real measurements and post them here.</span></p><p><span class="redactor-invisible-space">On the TX side I've also applied this power saving scheme but I had to apply less "off time" to avoid having to press the button a long time for the receiver to acknowledge the signal. It's about 33% off time. but I'll also try to lower the TX power so I can get aditional savings. </span></p><p><span class="redactor-invisible-space">TX: ~8mA (radio) + ~1mA CPU = ~9mA<span class="redactor-invisible-space"><br></span></span></p><p><span class="redactor-invisible-space"><span class="redactor-invisible-space">A CR2032 battery has around 240mAh of capacity so this results in 26hours of remote module autonomy.</span></span></p><p><span class="redactor-invisible-space"><span class="redactor-invisible-space">Assuming a average button press of around 3 seconds (will be lower than that but still) this gives an autonomy of 31200 button presses. </span></span></p><p><span class="redactor-invisible-space"><span class="redactor-invisible-space">No problem here, I gess :)</span></span></p>
                                
                            </p>
                        </li>
