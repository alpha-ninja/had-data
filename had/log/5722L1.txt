                            <h2><a href="/project/5722/log/19892-visited-maker-fair-got-tips-submitted-pull-request">Visited Maker Fair, Got Tips, Submitted Pull Request</a></h2>
                            <small class="authorship">2 months ago  &bull;
                                <a class="gray-link" href="/project/5722/log/19892#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-19892"><p>Hey! So I've been working to make a few revisions to the existing specification (<a target="_blank" href="http://retrospec.cc">http://retrospec.cc</a>), while adding a few connectors to the fray. I aim to make as few revisions to the spec as possible. People should be able to expect a rigid specification as they build. These revisions are directly related to feedback from fellow engineers.</p><p>I realized one thing in particular: there needed to be a connector geared toward those first trying out the spec. I decided on the DA-15 connector, since it is familiar yet slightly rare. The more 'pro' pins, in the new proposal, are found on DE-9 &amp; DB-25.</p><p>Others communicated their desire for a longer-distance communication protocol present on these connectors. While generic RS-232, RS-422 &amp; RS-485 were in the running, I decided on CAN bus. The bus will ideally provide the communication backbone for each group of modules. Consider CAN for both flexibility &amp; long cable runs, and SPI/I2C for the simplicity &amp; shorter cable runs.</p><p>There was another suggestion related to higher-current applications (i.e. motor controllers). There are connectors in the DB-25 form factor that feature a few large pins. These connectors are usually available via an online order and a slow boat. The connectors are a variant of older connectors which featured one or more coaxial connectors amongst the D-Sub pins.<br></p></div>
                            </div>

This video demonstrates some of the possibilities provided via this framework. In building the modules, I discovered pros &amp; cons of varying connector specifications. As things became more stable, I found I was able to deploy quite a few modules around the house, shop &amp; truck. Just the beginning...</p><iframe style="width: 500px; height: 281px;" src="//www.youtube.com/embed/gzQk8-3atvs" allowfullscreen="" frameborder="0" height="281" width="500"></iframe></div>

                            <h2><a href="/project/5722/log/23236-the-last-major-revision">The Last Major Revision</a></h2>
                            <small class="authorship">2 days ago  &bull;
                                <a class="gray-link" href="/project/5722/log/23236#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-23236"><p>Maker Faire Bay Area 2015 and the events leading up to it really helped refine the specification in a few very important ways. First, I found many of the features packed into the DB-25 spec should be moved to DA-15. This was in an effort to make the spec more accessible as one gets started. The DA-15 spec is not the best candidate for a plethora of chained modules. If chainability is your main focus, shift over to DB-25 and/or DE-9.<br>The old version of the spec featured USB and lacked CAN. USB is difficult to support when chaining is considered. USB is also a riskier thing to mess around with, since it probably has an expensive computer connected to it. CAN, on the other hand, chains easily and can be run over long distances. It is not as fast, but is sufficient for many electronics experiments. It is used extensively in automotive, automation &amp; robotics environments.<br>Finally, after taking a look at some other multi-pin connectors, I found it wouldn&apos;t be to difficult to support 1080i HD video. While not the cleanest means of transmitting video, it is an ideal transmission method for those who like to tinker. The other connectors I mentiond combined composite, &apos;super&apos; video &amp; component video onto three pins. This pattern has been followed with the <a target="_blank" href="https://github.com/makerblueprint/retrospecification/blob/incoming/connector-collection/d-subminiature/db-25.yaml">DB-25 connector pin spec</a>.</p><p><img src="https://cdn.hackaday.io/images/328471439789445302.jpg">When I decided to make an appearance at Maker Faire, I knew I would need a set of modules to help attract interest. I chose to embed a number of modules into an old flatbed scanner -- along with an Arduino Uno to run everything. The control surface featured all sorts of buttons &amp; fiber-optic-backlit DE-9 &amp; DB-25 connectors. The buttons &amp; LEDs were driven by I2C port expanders &amp; the NeoPixels via the `pixel-bus`. When one (or a few) of the buttons were pressed, the pin holes corresponding to the button label(s) would illuminate. Since I already had a 16x8 I2C LED display, I programmed the Arduino to display button label text there as well. The LED display was behind the scanner glass, while the control surface was mounted on a board well below. The scanner &apos;display case&apos; and other parts of the display were linked by a single 25 connector ribbon cable. There were other means of input &amp; output as well. Important to the design: I needed certain modules to protected, while others could be messed with. Those goals were well achieved.</p><p><img src="https://cdn.hackaday.io/images/1128371439789675903.jpg"></p></div>
                            </div>

                        <li>
                            <h2><a href="/project/5722/log/23259-revision-updates-new-parts-of-the-spec">Revision Updates &amp; New Parts of the Spec</a></h2>
                            <small class="authorship">2 days ago  &bull;
                                <a class="gray-link" href="/project/5722/log/23259#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-23259"><p>After the Maker Fair 2015 revisions were adopted, I went back to work updating a bunch of Retro Modules which were no longer in compliance. This is one of the reasons I&apos;m striving toward a stable pin specification in short order. It is not fun looking at a box full of no-longer-compliant modules. I&apos;ve had to sift through existing modules to find the few which were not affected by the spec revisions. Thankfully, I am gaining momentum again.<br>When I built out the hardware for my &apos;Christmas Truck&apos; project, I used the Retro Modules spec extensively. Those modules will need slight updates now, but even the basic modularity of the install proved useful. I recently traded my truck in for another, and the uninstall process was a breeze. That is -- everything except some of the oldest modifications I made to the truck. The 2012 Tacoma features a LCD screen in the rear-view mirror. It, at times, displays composite video from the stock rear-view camera. I wanted a way to send my own video to the mirror without making a big mess of the mirror housing. I was able to locate the video cable &amp; make a splice quite some time ago. To undo this splice, I had to break out the soldering iron &amp; solder the short segment of wire back together. If I decide to do a similar modification today, I would make it easy to remove any mods. How? I would add connectors (blade, etc) which allow the stock cable to be reconnected to itself with ease. This can increase the cost of the mod, but not by much.</p><p><strong>Headers</strong></p><p><strong></strong>I have spent some more time fleshing out pin header specs &amp; 
documentation. The very common 26 pin header is hardware compatible with
 a 25 pin DB-25 connector. The 10 pin header is compatible with the DE-9
 connector. This allows the use of old-fashioned header-to-connector 
ribbon cables. You may have had some of these cables in an older desktop
 tower.</p><p><em>&quot;AC&apos;97&quot;/&quot;HD Audio&quot;</em><span class="redactor-invisible-space"></span></p><p><img src="https://cdn.hackaday.io/images/6775761439794154086.jpg">Another common staple of desktop computing is the &quot;AC&apos;97&quot;/&quot;HD Audio&quot; header &amp; breakout modules. Many modern single-board computers feature analog &amp; digital audio, but are often broken out via consumer audio jacks. These small computers could leverage the AC&apos;97 header with great ease -- reducing circuit-board complexity &amp; increasing design flexibility. A source of confusion for many, however, is S/PDIF headers on motherboards. Legacy graphics cards often shipped with a tiny cable for routing digital audio. S/PDIF headers are not well standardized &amp; may consist of two to eight pins. Given this problem, I <a target="_blank" href="https://github.com/makerblueprint/retrospecification/blob/incoming/connector-collection/header/retro-audio-header.yaml">opted</a> to append five pins to the end of the &quot;AC&apos;97&quot;/&quot;HD Audio&quot; header. This keeps compatibility with off-the-shelf desktop audio modules yet allows S/PDIF in/out on a custom header if so desired.</p><p><br><strong>Pro audio is also an important area to focus on.</strong></p><p>Since part of this spec features analog audio next to other digital signals, there is a greater risk of noise. Many environments do not require pristine audio conditions. Some do. This is why I have included support -- in certain connectors -- for AES Digital Audio. S/PDIF is the consumer version of AES. AES42 has some really great features that quite a few sound engineers enjoy. These signals are generally sent on a standard 3-pin XLR connector. There is risk, though, in sending digital signals on a connector primarily used for analog audio. Audio mixers may get damaged if a AES signal is connected to an analog input. I have proposed <a target="_blank" href="https://github.com/makerblueprint/retrospecification/blob/incoming/connector-collection/pro-audio/xlr6.yaml">a spec for 6-pin IEC XLR connectors</a> to hopefully address this issue -- and add a feature or two. The 3-pin connector only allows for one-way transmission of digital audio signals (generally). A 6-pin connector allows for power &amp; bi-directional AES transmission. This opens up doors for chainable digital effects processors, digital audio pedals, mics &amp; remote monitoring. For instance, an artist performing on stage generally needs a mic, guitar input &amp; monitor mix. This could be accomplished via this single XLR cable. The monitor mix would arrive on the `digital-audio-aes-host-out-client-in` pins, and the audio source(s) on-stage would be sent back to the mixer via the `digital-audio-aes-client-out-host-in` pins. These on-stage modules could be powered via the `fourty-eight-volts-max` pin.</p><p><br><strong>This latest work can be seen on the incoming branch of <a target="_blank" href="http://retrospec.cc">retrospec.cc</a>.</strong></p></div>
