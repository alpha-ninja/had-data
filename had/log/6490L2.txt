                            <h2><a href="/project/6490/log/22442-monitoring-low-amplitude-outputs-via-ch3">Monitoring low amplitude outputs via CH3</a></h2>
                            <small class="authorship">8 days ago  &bull;
                                <a class="gray-link" href="/project/6490/log/22442#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-22442"><p>CH3 is specified as a +/- 3.3 V analog input with up to 32x programmable gain.<br><br>However, the schematic for the input signal processing stage looks like this</p><p><img style="width: 253px; height: 207px;" src="https://cdn.hackaday.io/images/4854681439243239825.png" height="207" width="253"></p><p>*A 10MOhm pull down resistor is also present on CH3, and is not shown</p><p>This functions as a unity buffer if Rg is left floating, and CH3 is calibrated under such circumstances. But if the user isn&apos;t satisfied with the +/-100mV range at 32x PGA, and wishes to obtain an arbitrarily high gain(subject to noise considerations) , then they may ground Rg via an externally plugged in resistor. The gain will now turn into the familiar formula for a non-inverting amplifier (1+Rf/Rg . Where Rf=10K , Rg=user configured)<br></p><p><strong>HB100 Doppler radar output monitoring</strong><br></p><p>Among many other uses, this functionality was used to add another 10x of gain to the 32x internal gain in order to monitor the output of an HB100 doppler radar.  The output of this radar is an analog signal whose frequency is a function of the velocity of the reflecting object.</p><p>Here&apos;s a video of the test</p><iframe style="width: 500px; height: 281px;" src="//www.youtube.com/embed/LFJo5rjPO9E" allowfullscreen="" frameborder="0" height="281" width="500"></iframe></div>
                            </div>

                            <h2><a href="/project/6490/log/22532-a-few-basic-electronics-experiments-with-guis">A few basic electronics experiments. With GUIs</a></h2>
                            <small class="authorship">8 days ago  &bull;
                                <a class="gray-link" href="/project/6490/log/22532#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-22532"><p><strong>Filter study</strong><br></p><p>A simple low-pass RC filter was made with a 10K resistor and 1uF capacitor.  The input was taken from the programmable sine wave generator, and simultaneously monitored via CH1 . The output(midpoint of R and C) was also monitored via CH2.<br>Frequency was swept from 10Hz to 500Hz, and the amplitude and phase plots are shown<br><br></p><iframe style="width: 500px; height: 281px;" src="//www.youtube.com/embed/YM6HixqZrAk" allowfullscreen="" frameborder="0" height="281" width="500"></iframe><p>Photograph of the setup:<br></p><p><img style="width: 478px; height: 271px;" src="https://cdn.hackaday.io/images/8213171439322682880.jpg" height="271" width="478"></p><p><strong>Diode IV characteristics</strong></p><p>A 4148 diode&apos;s IV characteristics are plotted.<br>Input is taken from a 0-3.3V DAC (PVS3) , and fed to a 4148 diode via a 1K resistor. The other end of the diode is grounded via a current to voltage convertor.  </p><p>IV plots are obtained by plotting the voltage across the diode versus the current</p><iframe style="width: 500px; height: 281px;" src="//www.youtube.com/embed/S9cLzfrC5Jw" allowfullscreen="" frameborder="0" height="281" width="500"></iframe></div>

                            <h2><a href="/project/6490/log/22534-testing-on-raspberry-pi">Testing on Raspberry Pi</a></h2>
                            <small class="authorship">7 days ago  &bull;
                                <a class="gray-link" href="/project/6490/log/22534#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-22534"><p>This project was originally envisioned as RPi2 specific, and prototypes were developed which included the 40-pin header on the vLabtool itself.  However, it was later decided to stick to USB, and make the software work with the RPi2 since it has 4 USB ports, and one can be dedicated to the vLabtool.</p><p>Some problems were encountered, and deemed necessary a few firmware changes.   The receive buffer of the RPi is much smaller than full blown desktops, and this caused overflows, and data loss.  The software has to bring large datasets after splitting them into smaller chunks.</p><p>No detailed testing has been carried out yet, and here&apos;s a picture of the oscilloscope GUI running on an RPi2.</p><p><img style="width: 732px; height: 422px;" src="https://cdn.hackaday.io/images/579911439325222855.jpg" height="422" width="732"></p></div>

                            <h2><a href="/project/6490/log/22651-calibrating-the-dacs-programmable-voltage-sources">Calibrating the DACs ( Programmable voltage sources)</a></h2>
                            <small class="authorship">6 days ago  &bull;
                                <a class="gray-link" href="/project/6490/log/22651#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-22651"><p>DACs have non-linearities that are repeatable, but are not usually periodic.  Hence, generating a single calibration polynomial is difficult (At least up to a few orders ).  A lookup table is the general solution for making an accurate programmable voltage source.  Read more about INL errors <a href="http://masteringelectronicsdesign.com/an-adc-and-dac-integral-non-linearity-inl/">here</a></p><p>The
 following image shows the deviation of one of the programmable voltage 
sources (resolution 1.6mV. +/-3.3V range) from the expected value(red 
trace) , and the accuracy of the same post-calibration(yellow trace ). 
The PIC has copious amounts of flash, and lookup tables and calibration 
constants can stored in it and loaded by Python upon initialization.</p><p><img src="https://cdn.hackaday.io/images/1693661435586142642.png"></p><p>The
 overall slope and offset is caused by the amplifying op-amp and 
corresponding resistors, and can be fixed with a second order polynomial
 easily.  </p><p>The jagged shape however, is a property of the DAC itself, and is corrected using a lookup table that stores the error in each code in terms of number of codes to skip forward/backward in order to achieve the highest resolution without compromising on accuracy.<br><strong></strong></p><p><strong>*As of this writing, the DAC calibration data is being loaded from text files for testing. They have not been written onto flash memory for the three units I&apos;ve shipped to SupplyFrame for the Best Product Prize.</strong><br></p></div>
                            </div>
