                            <h2><a href="/project/4927/log/15757-updated-internal-wiring">Updated Internal Wiring</a></h2>
                            <small class="authorship">5 months ago  &bull;
                                <a class="gray-link" href="/project/4927/log/15757#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-15757"><p><img src="https://cdn.hackaday.io/images/9400751427573864016.JPG"></p><p>Changed the USB sound card to a USB sound card with SPDIF. The FX3800 does not have it's own sound device like a Radeon. It has audio passthrough. It has an SPDIF connector &amp; what looks like an audio connector similar to those found on a cdrom, but this connector does not accept audio in like the SPDIF, it's for something else. </p><p>I am currently using the DVI connector from the GPU --&gt; HDMI in on the switch. I a working on dressing the cables, and my research indicates the best way to get audio out of this box is from the USB SPDIF --&gt; spdif in on the Quadro ---&gt; Display port out ---&gt; HDMI in on the switch. </p><p>To enable passthrough of the GPU, the onboard graphics, deep render, deep render standby, &amp; onboard HDAUDIO must be disabled. Therefore to have sound from the various VM's, we need some other sound device that can be passed through to the VM. SInce all USB's are passed through, we can use a usb sound card with SPDIF.. The other option would be:</p><p>1) more expensive Nvidia GPU with sound, which is certainly possible for some builds.</p><p>2) Radeon GPU with sound, but nearly impossible to get Android up on</p><p>3) Mini ITX board with additional mini PCIE, like the WiFi version of the spec board.  Then a mini-pcie sound card with SPDIF.  - IF the board supports a not WiFi card in this slot. </p><p>The USB-&gt;SPDIF is $14, so this is the solution for now.....We'll have to see what it sounds like.....<br></p></div>

                            <h2><a href="/project/4927/log/16059-vmware-licensing">VMware Licensing</a></h2>
                            <small class="authorship">5 months ago  &bull;
                                <a class="gray-link" href="/project/4927/log/16059#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-16059"><p>Just to clarify - we are using the FREE version of VMware's VSphere Hypervisor. </p><p>When I first discovered this feature, I called VMware to nail this question down for myself. I suggested a $99 version that included access to the API's only. No Vmotion, Vcenter etc. </p><p>They didn't see the value. </p><p>You can do what I'm doing with the free version, without the API's. Either method works, and the better or easier way really depends on what you are used to or comfortable with. </p><p>You can do ANYTHING over SSH, so I am using that. </p><p>BUT, to get the FREE version, you need to create an account. then DL the ISO and you have access to the FREE version KEY. </p><p>I assumed they would want at LEAST all the emails so they could send out promo's, etc</p><p>So I engineered this whole convoluted process where an end user would receive his/her Hydra, boot from a live cd, sign up and DL the ISO, get the KEY in email, and install to a virtual machine using the SSD as a physical disk. Then remove the CD and reboot to ESXi. </p><p>I forget what the exact roadblock was, maybe I thought the windows live cd would get me in trouble. So I called VMware again and after getting passed around a little I was able to ask them directly how to proceed. </p><p>"just install the FREE version and apply the KEY, the KEY is the same for all FREE installs. "</p><p>OK then. </p><br></div>

                            <h2><a href="/project/4927/log/16485-did-some-controller-development">Did some controller Development</a></h2>
                            <small class="authorship">4 months ago  &bull;
                                <a class="gray-link" href="/project/4927/log/16485#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-16485"><p>My Son Gregory is kind of in charge of the Controller Ergonomic design. He was here this weekend, so between that and the mouse pointer Hacklet we were inspired to some brainstorming.  We long ago decided a Trackball is a MUST, and the GameCube controller seems to be the one the consensus prefers.  We don't have one of those laying around, but we had a broken Xbox controller we could butcher up, just to see how it feels in your hand.  The big hole on the right would presumably have a Trackball instead  of what M$ currently provides. </p><p>This is our prototype butcher job:</p><p><img src="https://cdn.hackaday.io/images/8874971428952526051.jpg"></p><p>So imagine something similar to a GameCube Controller spread out as this is. The spot below the phone could hold the buttons that were removed to accommodate the phone.  It seems to feel OK, &amp; the similar to GameCube Controller would seem to have more room. </p><p>The Mouse Pointer Hacklet also inspired us, first of all to add <strong>mouse &amp; keyboard</strong> tags like we should have originally. </p><p>This project mentioned in the Hacklet interested me:</p><p><a href="https://hackaday.io/project/4340">ORB – A 3D gaming mouse.</a></p><p>Since he has the Arduino identified as a HID device.</p><p>  The Arduino in our Game Console only controls relays at present, and only gets power from the USB connection. The way way we currently provide the Mouse &amp; Keyboard is with the VNC server built into the Hypervisor, since it's available to ALL VM's and even works upon boot so BIOS control is possible.  As I have noted someplace above, I would have liked to use a Bluetooth HID but the ability to do this depends on the stack present in the phone.  But now I have the idea to have the Arduino behave as a HID, and send the scan codes over bluetooth from the phone.  The libraries also include Joystick,  audio in &amp; out, &amp; lots more,  so it looks promising that all  the functions present on the phone could be passed this way. </p><p>This is most likely a "future development" aspect of the project, since we presently have a working, if not optimal, solution. I suppose a spare Arduino  for testing may be in order. <br></p></div>
