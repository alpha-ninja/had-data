                            <h2><a href="/project/4395/log/20210-beaglelogic-coming-to-debian-jessie">BeagleLogic coming to Debian Jessie</a></h2>
                            <small class="authorship">2 months ago  &bull;
                                <a class="gray-link" href="/project/4395/log/20210#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-20210"><p>I've been testing Debian Jessie with my LCD cape project and I've found it to work nicely with the 4.1 kernel. The sigrok software BeagleLogic distributes has not been updated as the glib version bundled with Wheezy was lower than what newer versions of sigrok required to compile, hence, instead of backporting the newer glib versions to Wheezy, I'd be moving the entire setup on to Debian Jessie now.</p><p>I have also set up and compiled the latest released 4.1 kernel for the BeagleBone Black. For those of you not aware of it, the BeagleBone Cape Manager is back in 4.1 with a bang, so I can add support for my BeagleLogic cape as well :)</p><p>Along with this I'm testing the work of my mentee in this year's Google Summer of Code with BeagleBoard.org who is working on porting the remoteproc functionality to the newer kernels.<br></p><p>BeagleLogic would still run on the 3.8.13 kernel for a whille until the framework is at a point where I can look at porting it to the 4.1 kernels. These changes will happen over the next months after which you can expect a <strong>BeagleLogic edition</strong> system image with the software installed in package form (apt-get updates), just flash it on the SD Card, put on the cape and happy debugging!</p><p>And about the capes, I'm happy to have received responses from all over the world, so I think I can talk about the first batch of the assembled capes soon, so stay tuned!</p></div>
                            </div>

                            <h2><a href="/project/4395/log/22623-creating-a-system-image-for-beaglelogic">Creating a system image for BeagleLogic</a></h2>
                            <small class="authorship">6 days ago  &bull;
                                <a class="gray-link" href="/project/4395/log/22623#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-22623"><p>BeagleLogic is a collection of pieces of software that work together. For a new user it can be overwhelming to build all the components from the scratch.</p><p>As of now, I have been providing pre-built firmware for the PRUs and the sigrok binaries ( Debian Wheezy has an outdated library which breaks compilation of the latest sigrok libraries ) but they still have to be extracted into the system image. Also the software doesn&apos;t start up automatically and some scripts have to be executed to get BeagleLogic up and running. This might be acceptable, but isn&apos;t the best solution.</p><p>I&apos;m building a new system image that contains all components preconfigured to run BeagleLogic at startup, out of the box. This image is to be programmed into a SD card the same way as one would flash a regular image into a SD card.</p><p>For the moment, it is going to be restricted to &quot;git pull&quot; updates, but the idea is to bundle everything ultimately in a .deb package and submit it to the BeagleBoard Debian repositories.</p><p>Follow the logs for further updates.</p></div>

                            <h2><a href="/project/4395/log/23513-channel-naming-convention">Channel Naming Convention</a></h2>
                            <small class="authorship">2 days ago  &bull;
                                <a class="gray-link" href="/project/4395/log/23513#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-23513"><p>Here&apos;s the image for a quick reference.</p><p><img src="https://cdn.hackaday.io/images/3912841439839716272.jpg" width="311" height="462" style="width: 311px; height: 462px;"></p><p>Note that:</p><ul><li>Channel &quot;x&quot; is bit &quot;x - 1&quot; in the channel binary data (which is stored little endian) Right to left is channel 14 to Channel 1 ( ie bit 13 to bit 0)</li><li>Default data captured is in 8 bits, but can be set to capture 16 bits. See <a href="https://github.com/abhishek-kakkar/BeagleLogic/wiki/sysfs-attributes-Reference" target="_blank">sysfs attributes reference</a> for a complete list of parameters that can be changed.</li><li>The label alongside each channel pin is the name to be used while using the &quot;sigrok-cli&quot; app. So if you wanted to capture channels 1, 2 and 3 you would use the names P8_45, P8_46 and P8_43 respectively and not refer to them as 1,3 and 5. </li><li>However if you process raw binary data, you must use the bit numbers ( channel number - 1) with sigrok to receive correctly.</li></ul></div>
                            </div>
