                            <h2><a href="/project/4758/log/15166-finding-first-stage-proof-of-concept-components">Finding first stage proof of concept components</a></h2>
                            <small class="authorship">5 months ago  &bull;
                                <a class="gray-link" href="/project/4758/log/15166#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-15166"><h2>Proof of Concept / Prototype</h2><p>At the moment I am preparing to build a prototype for the sensor and receiver stations.</p><p>Although later builds may change based on power consumption, etc., to start with the system will be built with the following (mostly because I have them on hand and they are small)</p><p>The first prototype will have minimal sensors, etc., so we can experiment with wireless range and battery life<br></p><p><strong>Ground Sensor</strong><br></p><ul><li>Digispark</li><li>ESP8266 (later, may look at using just this using the gcc kit. But I want to get a working overall system first)</li><li>Power control circuitry, to turn on/off the ESP8266 when not needed to conserve power</li><li>1.5V to 5V converter</li><li>Small solar cell scavenged from a garden light</li><li>Charging circuitry</li><li>An LDR, used for scheduling attempts to send data to Repeater</li><li> DS18B20 waterproof temperature sensor</li><li>Various scavenged plastic / metal structure (look into 3D printing / fabrication of better design later)</li></ul><p><strong>Repeater Station</strong></p><ul><li>Arduino (Micro/etc)</li><li>ESP8266</li><li>DHT-11 (humidity &amp; temperature)</li><li>Larger solar cell, battery &amp; charging circuitry</li><li>A serial connection to a wireless router to connect to my home network, to simulate the long shot back to the farm</li></ul><p><strong>Home Base</strong></p><ul><li>A PC running Linux with a database to log everything</li></ul></div>

                            <h2><a href="/project/4758/log/15211-early-efforts">Early efforts</a></h2>
                            <small class="authorship">5 months ago  &bull;
                                <a class="gray-link" href="/project/4758/log/15211#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-15211"><h2>A bit of background</h2><p><em>Possibly some of what I'm talking about here may be a bit basic for many hackaday regulars. <em>I am an experienced software engineer &amp; hacker but have not had as much time as I would like to have played with</em> the micro end of embedded, so I am documenting experiences that are either new to me, or I am very rusty in. Recording my experiments may also help other new users.</em></p><h2>Early Prototyping Efforts (hardware Hello World)</h2><p>I will be building various prototypes using the digisparks even if a more evolved sensor design requires something else eventually.</p><p>The only arduino I had previously used to any extent was the leostick. I had purchased a comple of <a href="http://digistump.com/wiki/digispark/tutorials/connecting">digispark</a> last year (and a lot of other unfotunately sitll unused gear) so today I finally had a chance to find my feet with them.</p><p>Any competent software engineer will tell you that a very important task in any new software project is to build a hello world, to exercise the toolchain and the source code management systems, and sometimes overlooked, keep it maintained. I figure this is probably just as important even for something relatively trivial like an arduino based project. So I have started a <a href="https://github.com/pastcompute/hdp15">github</a> for my efforts already.<br></p><h2>Running on older PC hardware</h2><p>Rather than dragging around laptops I have an older desktop configured, a SFF case with a now-venerable VIA EPIA-5000 fanless motherboard. I built this up running Debian Squeeze 486 from a compact flash card, so it is a noiseless computer, which is great, because my lab is out the back and is a bit closer to nature than my office, I can hear the wind and birds and wildlife, etc. Anyway I got the digispark IDE which is essentially the Arduino IDE with digispark hardware running, and after fixing the <a href="http://digistump.com/wiki/digispark/tutorials/linuxtroubleshooting">usual permissions</a> issues still had a problem, an <em><strong>Illegal Instruction</strong></em> error.  This had be stuck for a bit then it clicked that my hardware is actually not even capable of running all 686 code... so after rebuilding the micronucleus driver and digispark avrdude things finally worked and I had a flashing LED / hello world.</p><p>One quirk with the digispark is you need to keep un/re-plugging it before re-flashing or the upload fails.</p><p>I then had to add in serial debugging by lashing together <a href="http://digistump.com/wiki/digispark/tutorials/debugging">a couple of resistors and a diode</a>, this worked quite well. (One reason I still have the old PC in the lab is it has two proper RS232 ports...)</p><p>Finally I hooked up a DHT-11 temperature and humidity sensor, using a <a href="https://github.com/markruys/arduino-DHT">low-memory no-frills DHT</a> library. I verified this was working with a very brief blast frmo the heatshrink gun. So far, so good...</p><p><img style="width: 245px; height: 183.645px;" src="https://cdn.hackaday.io/images/233361426423765566.jpg"></p></div>

                            <h2><a href="/project/4758/log/15293-early-lessons">Early lessons</a></h2>
                            <small class="authorship">5 months ago  &bull;
                                <a class="gray-link" href="/project/4758/log/15293#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-15293"><p>The digispark only has 8kb of flash, of which 6kb is available to sketch code. I should have realised, but it turns out I could only just fit serial debugging, DHT-11 and turning on the ESP8266 in AP mode. This might still be fine for a sensor with serial debugging disabled, but I also struck various electrical issues.</p><p>For 7 times out of 10 after flashing, even making sure I isolated the digispark to flash and then reset everything, I noticed that even though I could see the LED indicating the esp8266 was receiving the commands, the SSID was not showing up on my wifi scanner.  When I enabled serial debug the code reported no response bytes received from the esp8266 when that happened. I think it is a reset order / powering up problem or even some kind of ground looping issue, perhaps the pins in the digispark send junk down the serial line.  I noticed for example that if I left the digispark tx pin connected to the esp, flashing would fail.</p><p>To dig deeper I might have to drag out my storage cro and see whats happening, and/or put a jumbo sleep at boot so I can turn on the esp8266 manually after the digispark boots, with time to space.</p><p>But what all this really means, apart from being a learning exercise, is perhaps I need to consider some other processor even for the sensor. Perhaps its time to dive direct into bare metal code on the esp8266...</p><p>Options:</p><ul><li>use something other than the digispark for sensor processor. Pro: more flash; Con: maybe larger footprint, although an Arduino mini wouldn't be too bad</li><li>go direct to ATtiny85 bare C (or assembler!). Pro: fit more into flash; con: no nice arduino IDE or quick flashing</li><li>use something other than the ESP8266 for sensor telemetry. Con: need to find &amp; purchase more things, also maybe larger footprint, also, if not wifi, adds complexity</li><li>Avoid electrical issues by using bare metal C on the esp8266. Pro: reduces footprint even further; Con: no nice arduino IDE or quick flashing, and only two GPIO, so would need an I/O expander, which complicates things. This may be a downstream iteration anyway; having the Arduino IDE makes it easy to prototype connections to all  all the sensors &amp; confirm they are working</li></ul><p>First though I'll try and control the ESP reset from the digispark, and I'll probably spend a little more time cleaning up my wiring and checking the various lines are properly pulled up, power isolated and buffered and enough current supplied, etc. before deciding what to do. Maybe it is just something like a capacitor is needed or a common supply.</p><p>I don't want to waste too much time on this basic stuff, my available time will be limited as I have other things to do...<br></p></div>

                            <h2><a href="/project/4758/log/15441-native-esp8266-need-to-upgrade-to-esp-12">Native ESP8266, need to upgrade to ESP-12</a></h2>
                            <small class="authorship">5 months ago  &bull;
                                <a class="gray-link" href="/project/4758/log/15441#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-15441"><p>I have a very successful run managing to program and use the ESP-01 module standalone.</p><p><em>(I actually completed all this by yesterday, then today <a href="http://hackaday.com/2015/03/18/how-to-directly-program-an-inexpensive-esp8266-wifi-module/">this article</a> about natively programming the ESP-8266 came out on the main hackaday feed! As it turns out I took a slightly different approach, which I think long term will work better for me anyway)</em></p><p>I started with the <a href="https://github.com/nekromant/esp8266-frankenstein">frankenstein</a> firmware by <a href="http://hackaday.io/necromant">necromnt</a>, which uses the <a href="https://github.com/nekromant/antares">Antares</a> kconfig-based build system to build a uboot-like environment for the esp8266. I was quite easily able to add a DHT11 module, the results are now in the latest git for that project. I was actually quite impressed at how quickly I could get everything going, compared to other frameworks I have used.</p><p>The next step I did then was to strip down frankenstein to a <a href="https://github.com/pastcompute/hdp15/tree/41a1e6dbef547095bc0b01739746f1b3210f792c/ground-sensor/esp8266-bare">bare hello world</a> program, essentially ending up with just a  user_init() function with a few extra bits, but now with a nice build system that just seems to work: the deployment cycle is now simplified to two commands:</p><blockquote>make<br>make esp8266-deploy<br></blockquote><p>The only thing that didn't work was deep sleep mode (<a href="http://hackaday.io/project/3253/log/11726-deep-sleep">documented</a> here by <a href="https://hackaday.io/hacker/32012-matt-callow">Matt Callow</a>).  Turns out to make it work on an ESP-01 you need to hack of an LED and do some other <a href="http://hackaday.com/2015/02/08/hack-allows-esp-01-to-go-to-deep-sleep/">hardware mods</a>.  Not that I'm averse to doing that, but I want to design the sensor to be  easily constructed...<br></p><p>Functional deep sleep mode will be required to ensure I can run the sensor on a small solar cell and AA rechargable.  In the meantime, I'll code up a version that just uses a timer. It will be interesting to try and empirically determine the power usage differences.</p><p>So now on my shopping list is an ESP-12, a breakout with all I/O pins exposed...<br></p></div>
