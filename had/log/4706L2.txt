                            <h2><a href="/project/4706/log/19661-why-were-the-uv-badge-sensors-changed">Why were the UV-­badge sensors changed?</a></h2>
                            <small class="authorship">2 months ago  &bull;
                                <a class="gray-link" href="/project/4706/log/19661#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-19661"><p>The <strong>temperature, humidity and barometric pressure</strong> sensors have been changed to the <strong>B</strong><strong>osch BME280</strong> integrated environmental sensor.</p><p>Reasons:</p><ul><li>One sensor instead of two components, means lower cost, needs less space, and consumes less power</li><li>The energy hungry I2C interface has been changed to SPI</li><li>The BME280 provides absolute pressure sensing (while the former LPS25 needs calibration)</li></ul><p>In the original concept it was planned to 'estimate' the<strong> UV Index</strong> using an <strong>si1132 </strong>sensor. Long, torturous development lead to a divorce from this idea, however;</p><p>some reasons:</p><ul><li>si1132 measures visible light, then estimates the UV index; this can be very inaccurate in cloudy situations (according the datasheet as well)</li><li>The new sensor (ML8511) measures true UV radiation</li><li>The I2C interface should be dropped, as it's too energy hungry</li><li>The si1132 needs a complicated driver which is memory and energy inefficient </li></ul><p>More things about the <strong>si1132 operating method</strong>:</p><ul><li>The I2C interface is not static, it has a minimum clock frequency requirement...</li></ul><p style="margin-left: 40px;">Totally unacceptable! With a slower MCU, using software or semi­-software I2C interface this can't be used! Usually I2C and SPI interfaces are static, can be driven as slow as you want.</p><ul><li>The command operation expects a strange trial­error method. According to the datasheet you give a command, then you have to check whether it has been processed. If it has not been processed after several ms, it has to be repeated. <strong>What???? </strong>It's a very energy consuming process! Anyway we are talking about a <strong>digital interface</strong> which should process commands, but it acts more like a <strong>spoiled princess</strong>...</li></ul><p>So say good by to si1132 (sorry Silicon Labs...)</p><p><strong>Say hello to ML8511!</strong></p></div>
                            </div>
