                            <h2><a href="/project/7181/log/22873-project-logs-before-sleep-i-have-issues">Project logs before sleep... I have issues...</a></h2>
                            <small class="authorship">4 days ago  &bull;
                                <a class="gray-link" href="/project/7181/log/22873#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-22873"><p>Worked through some of the major software architecture &quot;snakes&quot; this week.  I have the basis of a communication protocol between the CANcrusher and the computer/tablet/whatever whether it&apos;s via USB or bluetooth.  One of the hurdles I wanted to deal with was how to buffer the transmitted data, how much to buffer, etc.  When CAN messages are flowing into the stream in 170-200us bursts and there are multiple CAN channels being monitored and sent off or logged, some of the packets have to be buffered.  Since I&apos;m using RawHID for the USB interface, I&apos;m limited to 64 byte frames.  I like using circular buffers for this type of thing, but I didn&apos;t want a single big buffer of all the frame bytes mashed into a single holding array and I didn&apos;t want some kind of rolling buffer of arrays.  Instead I ended implementing a circular buffer of array pointers, addresses to the head and tail buffers.  This seems to work well.  The goal (crazy as it sounds) is to have the basic structure of this concept working this weekend and implemented on both a simple Java app and in the CANcrusher.  We&apos;ll see how it goes...</p></div>
