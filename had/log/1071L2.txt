                            <h2><a href="/project/1071/log/10983-working-on-vsphere-60-beta">Working on Vsphere 6.0 beta</a></h2>
                            <small class="authorship">9 months ago  &bull;
                                <a class="gray-link" href="/project/1071/log/10983#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-10983"><p>First of all, before the Hydra was eliminated from the space trip contest, my attention was unfortunately entirely focused on income.  This has been back-burnered since late July, with not much forward motion since.  I didn't really expect to place or show, but entered for the publicity &amp; collaborative interaction it promised.  But at this point it's still just a hobby....</p><p>I downloaded 6.0 beta with the best of intentions, and it just sat in a folder, until Susie Felt from VMware called me and I asked her about PCI-Passthrough in 6.0, but she could not comment on it as it is still un-released.  So I had to try for myself.....</p><p>I have been installing Digital Menu Boards for McDonald's lately, and the Hydra certainly could drive those 5 HDTV's. Maybe not for McD's, but there are MANY others who have not converted from cardboard menus yet.  This would be a new market for VMware, and their interest would help to accelerate development.</p><p>I have been stuck on ESXi 5.0 build 768111 (this is 5.0 with some patches to fix VM auto starting) since ALL later builds break the ability to pass through onboard PCI devices. This made all the USB's unavailable, as well as any disk controllers (if I was inclined to try).  5.0 works, but how can I build a system that requires anything but the latest build?</p><p>In 5.1 they would revert back to not enabled for passthrough after the required reboot. </p><p>In 5.5 they would persist, and I could assign them to a VM, but the VM would not boot with these devices. </p><p>In 6.0 build 2159203 initial testing I have an XP VM using the AMD 4650 add in card, as well as 1 onboard USB.  I have no reason to expect anything less than the functionality I had in 5.0 with the patches. VMware has a "bug hunt" contest, so I expect basic features like autostart to work as well. </p><p>I will be trying the PCIe switch next.....<br></p></div>

                                <div id="post-body-21086"><p>So, I got an Oculus Rift DK2 cheap off CL. Tried for a solid day trying to get it running on the Hydra Game console, but no go due to the Quadro GPU.  The six headed Hydra here uses HD6450&apos;s, and it lit right up.  Just another peripheral, but 5 Oculus Rifts connected to 5 VM&apos;s seems to suggest some very cool 5 way gaming setups. Already have the Leap Motion working (in Ubuntu, with Compiz extensions you can spin the globe with your hand)  and  will be checking out some Leap + Rift games/content.  Any suggestions? </p></div>

                        <li>
                            <h2><a href="/project/1071/log/22084-android-vmapp-is-the-best-management-method">Android VM/App is the best management method</a></h2>
                            <small class="authorship">16 days ago  &bull;
                                <a class="gray-link" href="/project/1071/log/22084#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-22084"><p>VMWare ESXi being a Datacenter Hypervisor, the management available from VMWare is the VSphere Client, which needs to run from another PC. Or from a VM on your host, but you still need another PC to get <em><strong>that </strong></em>up and running.  You can do it from the console, but probably not without another PC to Google yourself through it.  And the VSphere Client has many options an end-user does not need, and some an end-user should NOT have access to. </p><p>If this is ever going to be a end-user friendly system, I needed some other interface. A VM with the VSphere Client requires Windows, and a license.....So I can&apos;t &quot;include&quot; a mgmt VM without the whole OEM M$ etc....</p><p>So the VSphere Client is of course available from VMWare for those that are comfortable with it. But I needed a simpler management interface that would be always available without extra hardware, needs to communicate via SSH or Serial to send the vim-cmd commands to the Host, and needs to be able to VNC into a VM screen (for building or troubleshooting VM&apos;s).</p><p>First I was using an HP Lights Out card as the primary adapter.  You could remote in and mount ISO&apos;s, and power the system on/off.  The colors on the DCUI were all messed up, but it worked. </p><p>Then I wrote an Android App that would run on a cheapo tablet that I would include. It needed WiFi, so I ran a VM running a soft-AP using a Ralink USB.  They don&apos;t last very long, not surprising considering the source &amp; price ($4).  And when the radio dies, you&apos;re stuck with VSphere.  But it would VNC into a booted VM, so you could get out of &quot;startup repair&quot; if needed. </p><p>Then I wrote a Python Menu, as an installable VIB.  This always worked for enumerating existing VM&apos;s and controlling their power &amp; snapshots.  No VNC from the console,  even tried some frame-buffer VNC without success.  This code is probably useful somewhere though....</p><p>Then I found I could &quot;steal&quot; the primary GPU from the Host, and give it to a VM.  It would not work with the ILO card, so it took me a while to discover this.  The ESXi screen  freezes, and the VM using the GPU appears at the console when booted.  No text mode, so you can&apos;t troubleshoot a non booting VM from here. (so whatever I run here needs to be RELIABLE)  But, this is a PCI (not PCIe) card, and all PCI slots in the Host can only be assigned to 1 VM (so these slots are kind of useless anyway).  I have these populated with a Geforce FX5200 &amp; a USB card.  I connected a touch screen monitor and am running Android x86 ICS with these 2 PCI cards.  Now I can run the Android App from an autostarting VM on every boot.  A nice touch screen interface requires no keyboard or mouse. The VM with the App can be &quot;frozen&quot; at a working state and return to the snapshot on every boot. The same App could also be on a phone or tablet the EU already has - or not.  I could potentially modify the install ISO to include my App and run it live.  I could also put the cheapo Ralink in the free USB and run an Android Tether soft AP in case there is no existing AP.  The DCUI can be accessed over SSH, so that&apos;s not gone.  No M$ license needed either. This seems to be the best EU friendly management interface and I am finally satisfied with it.   </p><p>Now I&apos;m re-writing the App.....<br></p><br></div>
