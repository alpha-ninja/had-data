                <div class="section section-description">
                    <h5>Description</h5>
                    <div class="description">We used a Kinect (version 1) and created a program in Processing to find the location of an object. The real world problem we selected is motor disability as a result of stroke or other illnesses. Our project aim is to help those with motor impairment by interfacing a robotic arm with our program in Processing to report the X,Y position of a randomly placed object to the arm. <br /><br />Our code has since been updated to include communication with the robotic arm's open-source software (BCI2000). Now, when the user moves the arm (via their brain activity) to a radius surrounding the target (given by the X,Y position data), the arm will then center itself over the object and drop down automatically to pick up the object. </div>
                </div>

                            Details
                        
                    </h5>
                    

                    
                        <div class="post-content details-content">
                            <p><p><b><b>First and foremost, I wish to thank my fellow team members, all of which helped write the code, test the setup, and brainstorm ideas for improvement as we went along. The entire project from conception to implementation took approximately 1 month. </b></b></p><p><strong>TEAM:</strong></p><p><b><b>Michael Van Dyk who hel<strong>ped t</strong>remendously with the conceptual design and equations.</b></b></p><p><b><b>Jianjun Meng who setup communication between this program and the robotic arm software by writing separate code that delivered the corrected positional data to the arm (not for viewing in this project).</b></b></p><p><b><b>Shuying Zhang who helped sharpen the Kinect's localization accuracy and provided us with the detailed summary (including the diagrams) below.</b></b><br></p><p><b><b><br></b></b></p><p><b><b>INTRO:</b></b></p><p><b><b>In the application of Brain Computer Interface (BCI), we demonstrated the use of Kinect sensor to give the real-world coordinates of a target object and to inform the robotic arm to grasp the target accurately. The proposed method mapped a 2-D image to the real-world coordinate of an object by calculating distortion due to distance from the sensor. The object recognition relied on a color tracking function with use of the (red, green, blue) RGB camera on the Kinect sensor. The proposed method was able to provide reliable and accurate coordinates with of an object in a 12-inch by 12-inch square range. <b>The accuracy was estimated at 96% based on a 4% error rate.</b></b><br></b></p><p><b><b><b><br></b></b></b></p><p><b>DIAGRAMS / SCHEMATICS:</b></p><p><b><br></b></p><p><b>Microsoft Kinect v1 sensor:</b></p><img src="https://cdn.hackaday.io/images/4032471431445977856.png"><br><b><p><strong></strong><b><strong>Basic parameters for Microsoft Kinect sens</strong>o<strong>r:</strong></b></p><table><tbody><tr><td>     Field  of view (Horizo<strong>nta</strong>l x Vertical)   </td><td>     57°  x 43°   </td></tr><tr><td>     RGB  camera resolution (H x V)   </td><td>     640  x 480 (VGA)   </td></tr><tr><td>     RGB  camera color depth   </td><td>     24  bits (8 bits per channel)   </td></tr><tr><td>     IR  camera resolution (H x V)   </td><td>     640  x 480 (interpolated) and 320 x 240   </td></tr><tr><td>     IR  camera color depth   </td><td>     11  bits   </td></tr><tr><td>     Maximal  frame rate   </td><td>     30  Hz   </td></tr></tbody></table><p><b><br></b></p><p><img src="https://cdn.hackaday.io/images/3356111431448350915.png"></p><p><b><br></b></p><p><b>A schematic camera view of the square range. The origin starts from the upper right corner. +x direction is horizontal pointing the left and +y direction is vertical pointing down on the screen.</b><br></p><p><img src="https://cdn.hackaday.io/images/1641431431446945011.png"></p><p><b><br></b></p><p><b>Camera view of the square range. Note that the square looks like a trapezoid from the perspective of the camera. </b><br></p><p><b><br></b></p><p><b><img src="https://cdn.hackaday.io/images/607181431448196952.png"></b></p><b><b><p><b><b><br></b></b></p><p><b><b><b>The side view of the camera and the square range.</b><br></b></b></p><p><b><br></b></p><p><b><b><img src="https://cdn.hackaday.io/images/8612451431448403845.png"></b></b></p>A schematic screen viewed by the camera to calculate the x position of the object. The outer edges are the boundary of the screen. The size of the screen is 640 x 480 pixels. The coordinates represent the pixel number with respect to the origin located at the upper right corner. </b><br></b><p><b><b><br></b></b></p><p><b><b><b><b><b>-----------------------------------------------------------------------------------<b></b><b><b><b><b><b><b>------------------------------------------</b></b></b></b></b></b></b></b></b><br></b></b></p><p><b><b><br></b></b></p><p><b><b>CALCULATING X COORDINATE :</b></b></p><p><b><b><br></b></b></p><p><b><b>The x-coordinate of the object is obtained by calculating relative position of the object to the left and right sides of the trapezoid. The field of view we scanned was shortened in the code (from 640 x 480 pixels) to only scan through the pixels within the trapezoid (based on the coordinates we obtained through color tracking using the RGB camera on the Kinect). We chose to make the assumption that there was no distortion, which means that x2 and x1 appear equal in size to m1. This does not imply that there is no x distortion, since m1 is closest to the lens, an object placed there will appear larger than objects placed further away on either side. However, since we were scanning a small area of 12 x 12 inch square that was ~2-3 ft distance from the base of the Kinect, the difference in the x coordinates we measured along the x axis appeared negligible. The diagram below depicts the slight difference (1-2 pixels change) between x coordinates along the same x-axis:<br></b></b></p><p><b><b><img src="https://cdn.hackaday.io/images/6909211431449798678.png"></b></b></p><p>Since there is no x- axis distortion, the ratio of the object position to the left border of the square is equal to the ratio of that to the left border of the trapezoid. Based on this relationship, we were able to calculate the x-coordinate of the object. The first step was to collect the object's coordinates at all four corners in the 2D image and then to map the position of the object...</p></b>

                    <h5>Project logs</h5>
                    
                    <ul class="buillogs-list post-content details-content">
                    
                        <li>
                            <h2><a href="/project/4975/log/22367-20150227">2015.02.27</a></h2>
                            <small class="authorship">8 days ago  &bull;
                            <a class="gray-link" href="/project/4975/log/22367#discussion-list">
                                0 comments
                            </a>
                            </small>
                            <p><p>In an effort to understand why the tracker kept switching between what we perceived as the color red and the color black, we used a pick color function that takes an input of a mouse click over a particular color in the Kinect picture and it reports back that RGB value. <br></p><p>We clicked on the red object and the black object and noticed that depending on the location of the color away from the Kinect RGB lens, the reported RGB values would vary wildly and almost overlap the further away from the camera they were. </p><p>As we were getting discouraged about hitting this stumbling block, we started thinking about why this is happening. Changes in the lighting affects the color. So what about using a marker that doesn&apos;t change in RGB value because of light conditions in the room - light itself?</p><p>So then we inserted an led circuit inside our block with a switch to conserve power. We clicked on the lumen of the LED in the picture and saw it&apos;s reported value was 255, 255, 255 the color white. We set this as the color we wanted to track. We had to tilt the LED at an upward angle so that the Kinect would see it.</p><p>The accuracy jumped from 50% to 96% as the color values of the light would not change depending where in the scene it was. </p><p>Now the average of the reported X,Y location values from the Kinect are transformed through our functions in the code and are given to the robotic arm as real X,Y coordinates along a horizontal instead of vertical plane. </p><p>Someone just puts on the EEG cap, imagines moving left or right, up or down, and once the arm reaches a half-inch boundary surrounding the object for about 1-2 seconds, the arm knows to center itself above the center of the randomly placed object before it drops down to pick it up, all based on the information we are giving it. </p>
                                
                            </p>
                        </li>
                    
                        <li>
                            <h2><a href="/project/4975/log/22366-20150220">2015.02.20</a></h2>
                            <small class="authorship">8 days ago  &bull;
                            <a class="gray-link" href="/project/4975/log/22366#discussion-list">
                                0 comments
                            </a>
                            </small>
                            <p><p>Next, we thought alright...to keep it from tracking objects outside of the scene we wanted the robot arm to pick up the block in, we could also change the code for this. </p><p>So we added some parameters for the scanning through the Kinect picture. Instead of scanning through the whole picture and looking for the nearest value RGB values, only scan within the coordinate values of the boundaries of the table. We obtained these border values by placing the block on each corner and recording the Kinect&apos;s reported X, Y coordinates. </p><p>This seemed to solve all of our problems at first, up until we saw it switching between the red marker and a black dot marking the origin on our table. </p>
                                
                            </p>
                        </li>
                    
                        <li>
                            <h2><a href="/project/4975/log/22365-20150212">2015.02.12</a></h2>
                            <small class="authorship">8 days ago  &bull;
                            <a class="gray-link" href="/project/4975/log/22365#discussion-list">
                                0 comments
                            </a>
                            </small>
                            <p><p>After testing out the color tracking code, it seemed to work really well at first. This was especially true when we switched to a red colored marker that we stuck into the foam block. Then the tracker started bouncing back and forth between all of the red or even dark colored objects in the room (fire alarm on the wall is one example). </p><p>Thus we encountered our first technical issue.</p><p>At first we only saw the tracker occasionally going to this far away location, before it moved back over to the red marker. So we thought to include some averaging in the code and have it report the mean of a large amount of sample X,Y locations based off of the RGB color we wanted to track. </p><p>This worked a little bit better for the meantime. Though some days the tracker was way off and we knew it wouldn&apos;t work unless we were getting consistent results. </p>
                                
                            </p>
