                <div class="section section-description">
                    <h5>Description</h5>
                    <div class="description">Ya'll know the signs in subway stations that show the time until the next train arrives? This project aims to create a miniature version for home use so you know when to start walking to the station.<br /><br />The unit will have four 5x7 LED matrix display units to indicate the arrival time of the next two trains. The data will be fetched from a centralized server, so more cities/services can be added to the device without requiring a firmware update.<br /><br />Communication with the network will be over Wifi via an ESP8266.<br /><br />A stretch goal is to create a simple mobile app to facilitate configuring the device during first setup, and thereafter.</div>
                </div>

                        <li>
                            <h2><a href="/project/6696/log/20688-host-design-considerations">Host design considerations</a></h2>
                            <small class="authorship">a month ago  &bull;
                            <a class="gray-link" href="/project/6696/log/20688#discussion-list">
                                0 comments
                            </a>
                            </small>
                            <p><p>I'm in the beginning phases of thinking about the design of the second PCB that will host the main microcontroller and wifi radio. The physical PCB itself will use the same 43x70mm form factor as the display controller, so they can be easily fitted together.</p><p>All I really know for certain at this stage, is that I'll want to stay in the ARM Cortex world, as that is what I am most familiar with. Maybe a <a href="http://hackaday.com/2015/05/07/were-giving-out-125-teensy-lc-boards-this-week/" target="_blank">Teensy</a> would work well?</p><p>I will likely have the communication between the device &amp; both the time-server and control apps be raw TCP. That way, I don't have to concern myself with parsing stringy HTTP headers and storing large HTML strings somewhere.</p><p>Finally, I'm toying with the idea about playing around with the <a href="http://www.freescale.com/webapp/sps/site/prod_summary.jsp?code=MPR03x">MPR03</a> touch controller. I got a few samples from Freescale of this tiny (2x2mm!) 8-pin 3-channel touch sensor last week and have been looking for an application for them.</p>
                                
                            </p>
                        </li>
                    
                        <li>
                            <h2><a href="/project/6696/log/20685-display-board-designed">Display board designed</a></h2>
                            <small class="authorship">a month ago  &bull;
                            <a class="gray-link" href="/project/6696/log/20685#discussion-list">
                                0 comments
                            </a>
                            </small>
                            <p><p>I finished the schematic for the display controller board yesterday &amp; sent the gerbers to the manufacturer.</p><p>I am going with a two-board design for the first revision of Subwaymate, as it will help compartmentalize features and aid in avoiding silly design mistakes on the board from making coasters of the whole thing. Also, by keeping the display unit agnostic from the rest of the project, I will have a display to use on future projects.</p><h2>Controller primary components</h2><ul><li>4 5x7 row-anode LED matrices</li><li>2 Texas Instruments TLC59284 sixteen-channel LED driver (constant-current sink)</li><li>1 CD4022B 8-output counter</li><li>1 74HC540 inverting buffer</li><li>7 IRLML5203 P-channel mosfets</li><li>Freescale Kinetis MKL02Z32VFM4 Cortex M0+ MCU (32k flash, 4k (!) RAM)</li><li>So. Many. Interfaces.</li></ul><h2>Brief design overview</h2><p>This display is drawn row-by-row. The row to be drawn is selected by the counter, which feeds it output through an inverting-buffer, then out to the P-channel mosfets. The columns are selected by the TLC59284 which are essentially shift registers that can sink considerable (configurable) current, so you can avoid having many current limiting resistors for the LEDs.</p><p>The Kinetis MCU will be responsible for drawing the display, and receiving commands from the main controller. I have exposed the MCU's i2c, SPI and UART lines to headers.</p><p>The raw inputs for the row/column selection have also been broken out to the left side of the board - 4 pins are required (shift clk, in, latch + counter clk). This can be reduced by 1 if a solder bridge is used to tie the clock of the counter to the latch of the shift register.</p><h2>Software overview</h2><p>The software is still in the planning phases, but the rough plan is to use a slim RTOS like <a href="http://atomthreads.com/">AtomThreads</a> to both keep the display refeshed and take input from the host controller over i2c, SPI or serial.</p><p>The MCU will have an ASCII character set preloaded - but the command set will also allow the host to toggle single LEDs, rows and columns.<br></p>
                                
                            </p>
                        </li>
                    
                    </ul>
