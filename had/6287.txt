                    <div class="description">I love timelapses! But I've found it difficult to record them. Using a phone would be possible, but then I can't use the phone ...  DSLRs are also expensive and while using a cheap keychain camera from china with simple circuitry for the delay, the quality is horrible. <br /><br />I would like to make a 720p or even 1080p timelapse camera for under $15, so you could have multiple of them on different project, all recording.<br /><br />The basic idea is this:<br />I would Use a PSoC4 BLE or some kind of combination of a MCU + BLE (or BT). Bluetooth would be used for setting the delay for taking a picture and also for streaming live video from the camera to a phone or tablet.<br /><br />I would use the same camera that is in the iphone 4. It costs under 2 dollars from china and the Iphone 4 takes nice pictures.<br /><br />The trickiest part would probably be procesing the data from the camera and interfacing the camera with the MCU.<br /><br />Similar project: <a href='http://orbides.org/lapse_camera.php?lng=eng' target="_blank">http://orbides.org/lapse_camera.php?lng=eng</a></div>

                            <h2><a href="/project/6287/log/19823-thoughts-on-which-microcontroller-to-use">Thoughts on which microcontroller to use</a></h2>
                            <small class="authorship">2 months ago  &bull;
                            <a class="gray-link" href="/project/6287/log/19823#discussion-list">
                                0 comments
                            </a>
                            </small>
                            <p><p><strong>Abstract</strong>: Atmel SAM3S + ESP8826 seems to be my best bet.</p><p>I searched around a bit more. As I already knew, if I were to use a STM or NXP mcu, it would cost 10+ dollars for a mcu with DCMI (digital camera interface). And I would still need a chip for the BLE radio.</p><p>Today I found out, that <a href="http://www.atmel.com/products/microcontrollers/arm/sam-s.aspx">Atmel SAM S70</a> also supports a CMOS camera interface.</p><p><img src="http://i.imgur.com/zxR5rHB.png"></p><p>It looks neat, but also simple enough to implement with the logic blocks on the PSoC. At least the SAM S70 is cheaper (about 7 dollars(at 1k units)) than the NXP or ST chips, and also in a hobbyist friendlier package.  </p><p><a href="http://valentfx.com/gradient-filter-implementation-on-an-fpga/">This webpage</a> will also be very useful if I'll take the PSoC + camera interface with logic blocks approach. <a href="http://www.cypress.com/?mpn=CY8C4127LQI-BL473">At 2.29 dollars (at 1k units)</a><a href="http://www.cypress.com/?mpn=CY8C4127LQI-BL473"> </a> it would seem as the cheapest option, but that option comes without DMA or Universal Digital Blocks<span class="redactor-invisible-space"> so it is useless for what I am doing. The ones that actually have the UDBs cost around 5 dollars.</span></p><p>Later, I found <a href="http://www.atmel.com/Images/doc11091.pdf">this application note from Atmel</a>, where they use a SAM3S4C mcu which costs only <a href="http://www.digikey.com/product-detail/en/ATSAM3S4CA-AUR/ATSAM3S4CA-AURTR-ND/4119480">4.6 dollars (at 1k units)</a>, which is really nice.</p><p><img src="http://i.imgur.com/cOVcbuM.png"></p><p>And probably the best thing about it is that atmel has already written some examples with JPEG compression which is just fantastic.</p><p>Maybe I would use WiFI instead of BLE? It seems easer to implement but more expensive to build. Than again, an ESP8826 does not cost that much.</p>
                                
                            </p>
                        </li>
