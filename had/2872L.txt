                            <small class="authorship">a year ago  &bull;

                            <div style="position:relative">
                                <div id="post-body-9220"><p><strong>*Synergy:</strong> Please visit and donate to synergy-project.org. <a href="http://www.synergy-project.org/">http://www.synergy-project.org/</a></p><p>First I would like to explain my love and addiction for a favorite program called Synergy. It shares a keyboard and mouse between multiple computers. One computer with a keyboard and mouse is set as a Synergy server, then other computers are added as clients and communicate over a local network. I love that its Open Source, multiplatform, and has a quick smooth transition from one screen to another.</p><p><strong>Synergy dislikes:</strong><br></p><ol><li> Its network dependent so any computer or router with connection problems fails.</li><li> All client keystrokes like passwords and account numbers go through the local network, although there is an option for encryption.</li><li> Setup can be complicated with mixed platforms.</li><li> It can not be used on screens where the client is not working yet, or disabled. For example; BIOS setup, Windows user account control, live CD&apos;s, some network passwords and user logins.</li></ol><p>I still like, use and highly recomend Synergy. This projects device is an all hardware solution that eliminates Synergy and the network.</p><p><strong>HID Mouse:<br></strong>This device is also focused on adding mouse features on the device hardware itself (touch sensors, sliders, key switches, optical encoders,) in place of features provided by drivers.</p><p>The USB input marked HID Mouse makes any mouse a driverless HID device (USB Host to serial), giving control to the microcontroller as a HID.</p><p>Drag along Keyboard:<br>When the screens switch, the microcontroller switches a USB switch, which connects the keyboard to the right computer. The delay for connecting to the host computer should be short considering it is just the keyboard.</p><p><strong>Desktop Integration:</strong></p><p><img src="https://cdn.hackaday.io/images/3502011439718552060.jpg"></p><p>This is a mixed mind map flow diagram showing how screen switching is 
done by detecting the mouse pointer hovering at a screens edge.</p><p><img src="https://cdn.hackaday.io/images/8827481409089380458.jpg">These are the chips I would like to use to make a board for production:</p><p>(1) ATmega32U4 Microcontroller<br>(4) AT43301 Low-cost USB Hub Controllers<br>(4) MAX3421E USB peripheral/host controllers</p><p>But I am going to make an advanced prototype out of other circuits for now, because this one is for me.</p><p>This is what I would like it to look like:</p><p><img src="https://cdn.hackaday.io/images/3500431409089589949.jpg"></p><p><img src="https://cdn.hackaday.io/images/4567011409089603787.jpg"></p><p><img src="https://cdn.hackaday.io/images/2835311409089613289.jpg"></p></div>

                            <div style="position:relative">
                                <div id="post-body-17004"><p>I have been gone for a while, sorry. I finally got a spot to set up my desk and computers, and I realized I had all the parts I needed to piece something together. </p><p>I seriously want to make a real circuit board and make this project legitimate... Oh crud, I dead bugged it again... I thought this was a work around for cost, but forces me to realize my fear of asking for help.</p><p><img src="https://cdn.hackaday.io/images/9919031430115320135.jpg"></p><h2>The Major Problems</h2><p>My original brainstorm, to count pixels and determine mouse location before and without communication with a computer, turns out to be more storm than brain. I have come up with two solutions to this, the first being the working version 1.0 prototype that cheats.</p><p>The second solution will be a long term experiment (with the pixel counting, or something), and a more powerful processor, possibly Intel based with Linux. This would keep my my other concept going, that it is possible to replace computer software drivers with an external hardware module system. The problem with the second solution is keeping the cost down. It will eventually get its own project log.</p><p>Getting this to work as intended is in need of a crude USB hack, that tricks the host computers into thinking the devices on this hub never get disconnected, even though they are switched around. My limited bubbled search leads me to believe it is an impedance hack, but there might be more to it then soldering four resistors on the USB output connectors.</p><h2>Version 1.0 Prototype Cheats:</h2><p><strong>Cheat 1.</strong> Use the Teensy touch sensors with thin wire and sticker like buttons, in any or all of the four combinations:</p><ul><li>A. At the sides of a dedicated mouse pad for left right screen switching.</li><li>B. Stuck on to a mouse as extra buttons for left right screen switching.</li><li>C. At the top of a dedicated mouse pad for 1 through 4 instant switching.</li><li>D. Stuck on four monitors for 1 through 4 instant switching.</li></ul><p><img src="https://cdn.hackaday.io/images/7358741430095431661.jpg"></p><p>:</p><p><strong>Cheat 2</strong>. A Python Script (not working yet) on each computer that detects cursor edge position, then signals for the Teensy to do the switching through USB Serial. I know this really breaks my rule of No Drivers, and probably many other rules of common sense in programming, but it will look cool :)</p><p>All I have so far is for Windows only in the Python Shell, and where "yay left" and "yay right" = missing Teensy USB Serial code.</p><p>Python 3.4.3 32bit Win7 with pywin32 from: <a href="http://sourceforge.net/projects/pywin32/?source=navbar" target="_blank">http://sourceforge.net/projects/pywin32/?source=navbar</a></p><pre class="hljs python"><span class="hljs-keyword">import</span> win32api

screenx, screeny = <span class="hljs-number">1279</span>, <span class="hljs-number">1023</span>
x, y = <span class="hljs-number">1</span>, <span class="hljs-number">1</span>

<span class="hljs-function"><span class="hljs-keyword">def</span> <span class="hljs-title">detect</span><span class="hljs-params">(x, y)</span>:</span>
    <span class="hljs-keyword">while</span> x &gt; <span class="hljs-number">0</span> <span class="hljs-keyword">and</span> x &lt; screenx:
        x, y = win32api.GetCursorPos()
    <span class="hljs-keyword">return</span> x, y

x, y = detect(x, y)
<span class="hljs-keyword">if</span> x == <span class="hljs-number">0</span>:
    <span class="hljs-keyword">print</span> (<span class="hljs-string">"yay left"</span>)
<span class="hljs-keyword">if</span> x == screenx:
    <span class="hljs-keyword">print</span> (<span class="hljs-string">"yay right"</span>)</pre><p>I would definitely need a Linux and Mac solution also (help!)</p>Q? Would putting in a delay (from time import sleep, sleep(0.1)) make it less taxing on the CPU?<p>:<br> </p><p><strong>Cheat 3.</strong> I hacked it from other working devices that I had laying around... </p><p>"Oh crud, I dead bugged it again"</p><p>I used two 4pin headers as a floating socket so the Teensy is removable.<img src="https://cdn.hackaday.io/images/6883731430095945073.jpg"></p><p>LED's poke out the front show which port is selected. Another 4pin header also pokes out revealing four touch pins from the Teensy.</p><p><img src="https://cdn.hackaday.io/images/8954831430096122313.jpg"></p><p>I used the relay and diode from my mouse project.</p><p><img src="https://cdn.hackaday.io/images/2449881430116138997.jpg"></p><p>Back view:</p><p><img src="https://cdn.hackaday.io/images/923871430096953990.jpg"></p><p>More pictures, USB cables, and hopefully a video of it all working will be posted soon.<br></p></div>

                            <small class="authorship">3 months ago  &bull;

                            <div style="position:relative">
                                <div id="post-body-17608"><p>I would force it to act like a mouse as part of an expandable system for my project.</p><p>Each Teensy-LC pictured below would use its built in USB HID mouse function, receiving mouse moves and commands from a Teensy 3.1 and USB Host.</p><p>I can't believe how perfect the timing is for this. I recently received a Circuits@Home USB Host Shield Mini as a gift, and already had it rigged together with my Teensy 3.1. The combination translates the USB data to serial and routs it to the right port.</p><p><img src="https://cdn.hackaday.io/images/9731731431179204623.jpg"></p><p>I could make a 3D printable case file that would include several sizes depending on how many ports you wanted to build, or maybe a snap together system that adds ports to the bus as needed.</p><p><strong>Here is the part I really like:</strong></p><p><strong></strong>I really do not want to trick the host computers with a hack, as I mentioned in my previous log. This is the part I really want: </p><p><img style="width: 100px; height: 209px;" src="https://cdn.hackaday.io/images/8672491431246111373.jpg"></p><p>Using a separate Teensy-LC for each computer will maintain a continuous connection, allowing for fast switching from the device and eliminating the need for python scripts.</p><p><strong>Data Flow:</strong><br>USB Mouse to<br>USB Host Shield to<br>Teensy 3.1 using <a href="https://github.com/xxxajk/spi4teensy3">SPI4Teens3</a> in the USB Host Shield Library, to<br>I2C (may change this, but there is a new <a href="https://github.com/nox771/i2c_t3#i2c_t3">i2c_t3</a> library made for Teensy3!)<br>To each Teensy-LC then output as USB HID mouse moves<br>Relayed to the right computer.</p><p>I Soldered back pins on Teensy 3.1 and pins on the host shield.</p><p><img src="https://cdn.hackaday.io/images/9937091431178075695.jpg"></p><img src="https://cdn.hackaday.io/images/9259241431178101970.jpg"><p><img src="https://cdn.hackaday.io/images/880051431178384252.jpg"></p>Temporally hacked together for testing. Sorry there are not more pictures :( I was going to start a different project for this, but the timing is so perfect to switch to this now.<p><img src="https://cdn.hackaday.io/images/4676741431178166004.jpg"></p>Some pics that helped me with setup. I used a 100uF cap, cut the VBUS power trace on the USB Host Shield, and added a USB Hub with a 5V power supply.<p><img src="https://cdn.hackaday.io/images/5364421431178508866.jpg"></p>I promise to get this more organized, with details, pictures and links, that will explain exactly how to get the USB Host Shield working.</div>

                            <h2><a href="/project/2872/log/19627-stickvise-rules">Stickvise Rules</a></h2>
                            <small class="authorship">2 months ago  &bull;
                                <a class="gray-link" href="/project/2872/log/19627#discussion-list">
                                    3 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-19627"><p>I had been using the  <a target="_blank" href="http://hackaday.com/2015/06/13/review-stickvise-needs-a-place-on-your-bench/">roll-of-solder-to-hold-a-circuit-board-down</a> trick forever until I won this <a target="_blank" href="https://hackaday.io/project/3287-stickvise-low-profile-soldering-vise">Stickvise</a> from the <a target="_blank" href="http://store.hackaday.com/collections/products-tools/products/stickvise">Hackaday Store</a>. <strong></strong>The Hackaday Prize
is fantastic!<strong></strong></p><p><img src="https://cdn.hackaday.io/images/8409481434481738191.jpg"></p><h2>Review conclusion:</h2><p>A
solid bench tool that exceeds expectations and will last for years.</p><p><img src="https://cdn.hackaday.io/images/4716011434481708138.jpg"></p>Unpacking it, I quickly made the mistake of underestimating its small size. It firmly holds a board in place, nice and flat, with a solid metal feel. It flips over to so you can check your work. The grove in the jaws is perfect for holding boards, and I like the idea of jaw upgrades.<h2>Here was the biggest surprise:</h2><p>I took it out of the package and within 20 seconds, I had a Teensy in it adjusted to the perfect tension I wanted. Then I found some baby 1x2 and 1x4 headers and decided to start a header farm.</p><p><img src="https://cdn.hackaday.io/images/194441435312699699.jpg"><img src="https://cdn.hackaday.io/images/5102741435313583125.jpg"> The new sockets work fine.</p><p><img src="https://cdn.hackaday.io/images/2056531435312747407.jpg"></p><p><img src="https://cdn.hackaday.io/images/1275651435312765335.jpg"></p><p>I had this briefly working before, then when I set it up again, I got 
nothing. I thought I had blew up a board and totally missed the blue 
smoke. I checked everything with a meter a hundred times, and 
re-soldered everything three times. This is what I came up with. <br><strong></strong></p><p><strong>This 
doesn't work:</strong><strong></strong></p><p><img src="https://cdn.hackaday.io/images/6003771435312838016.jpg"></p><strong>This does:</strong><p><img src="https://cdn.hackaday.io/images/1885831435312867058.jpg"></p><p>I am thinking its interference in the shape of the SPI wires, even though the boards are so close to each other. (Blue=3.3V, Green=Ground.)</p><p>Am I wrong?</p><h2>The differences:</h2><p><strong>Not working:</strong><br></p><ul><li>MISO crosses over MOSI</li><li>SS goes around MISO on the Teensy side.<strong></strong></li></ul><p><strong>Does work:</strong><br></p><ul><li>SS crosses over MOSI</li><li>MISO goes around SS on the USB Shield side.</li></ul><p>(This makes more sense now that I see it on paper.)</p><p>This is very scary for me because I am just starting to play with KiCad, and would hate to order a board that had this problem. </p><p>Would a board with a ground plane and no “crossed wires” help so much that the shape wouldn't matter? </p><p>Are there board designs that are less SPI friendly than others?</p><hr>I used LibreOffice Draw to map the pins and wires upside down.
<p>Rotated diagram matches wiring pictures above:</p><p><img src="https://cdn.hackaday.io/images/3589601435314234041.jpg"></p>I cut the VBUS jumper on the USB Host Shield that supplies power to the host connector, added a USB hub for power instead, and put in a 100uF cap, as recommended here:<br><a href="http://pjrc.com/teensy/td_libs_USBHostShield.html">http://pjrc.com/teensy/td_libs_USBHostShield.html</a><br>and here:<br><a href="https://www.circuitsathome.com/usb-host-shield-hardware-manual">https://www.circuitsathome.com/usb-host-shield-hardware-manual</a><h2>Teensy-LC<br></h2><p>Even though the Stickvise comes in handy most of the time...</p><p><img src="https://cdn.hackaday.io/images/7916561435313069787.jpg"></p><p>but I prefer to get the spacing right for pins here first.<br></p><img src="https://cdn.hackaday.io/images/2553601435313460414.jpg"><p><img src="https://cdn.hackaday.io/images/2165811435361299109.jpg"></p></div>
