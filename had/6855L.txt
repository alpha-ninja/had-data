                        <li>
                            <h2><a href="/project/6855/log/21474-update-2-code-overhaul">update 2: code overhaul</a></h2>
                            <small class="authorship">24 days ago  &bull;
                                <a class="gray-link" href="/project/6855/log/21474#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-21474"><p>firstly, i switched from the Adafruit SSD1351 lib to the ucgui lib, result: framerate a few times faster :)</p><p>then i thought: &quot;hey..  if i put in a delay of longer than 1000ms in my main loop, the clock doesn&apos;t update for that second (using protothreads, and that library doesn&apos;t have real interrupts)</p><p>then i came to an idea, namely: digital pin 9 as a pwm output to a analog pin, and once the analog pin is low, the clock checks if it needs to update, so basically, i&apos;m using a interrupt! :)</p><p> then i created an app in the ported microtouch framework, and found out that i can use system variables in the app&apos;s code, aka: a huge securiy leak(!), if i set a system variable to a wrong value, the watch might freeze, same goes for functions, like calling the main loop() in the application&apos;s loop (resulting in running the main loop in a main loop..)</p><p>one solution is to rewrite the ported microtouch&apos;s app framework code, but that&apos;s a bit too much work as of now, so i think i will go with an easier solution: adding a __ in front of every app&apos;s variable and function, that way they won&apos;t interfere with eachother. :) (though i need to get on rewriting the code sometime though.) (feel free to comment your solution!)</p><p>plus i started working on a homepage for the Zerowatch, and asked AtomSoft for a custom pcb (ncluding assembling), because that&apos;s way too complex for me, and my hands get all shaky when soldering (i also don&apos;t have enough place for a reflow oven)<br></p><p>lastly: i started using app inventor 2 and got my own messages working now, it notifies me when i get a text message :)</p><p>and most importantly: everything will be open source on release.</p><p>well, that&apos;s it for this update :)<br></p></div>
                            </div>

                        <li>
                            <h2><a href="/project/6855/log/21502-update-3-framebuffer-lots-of-apps-possible-now">update 3: framebuffer? + lots of apps possible now!</a></h2>
                            <small class="authorship">24 days ago  &bull;
                                <a class="gray-link" href="/project/6855/log/21502#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-21502"><p>ii have looked into creating a framebuffer for the zerowatch, and calculated the RAM usage it would take</p><p>128*128 pixels *8bit depth (8 bit unsinged int, aka 256 colors) = 16 kilobytes of RAM</p><p>128*128 pixels *16bit depth (16 bit unsinged int, aka 65535 colors) = 32 kilobytes of RAM</p><p>fun fact: the 256 colors  (8bits) is a;ready half of the mcu&apos;s RAM on it&apos;s own where 16bit depth consumes all of the mcu&apos;s RAM!<br>so there will be no framebuffer on zerowatch sadly...</p><p>however, if you want to, you can implement a 8bit depth one, but you&apos;ll have to lower the app RAM buffer to about 4 kb.</p><p>perhaps change somewhere in the future to a mcu with more RAM (perhaps an STM32f411)?</p><p>aside from that, i have started working on having multiple pages of apps in the shell, instead of one page max.<br>this allows up to 100&apos;s of apps on the system (if you can fit them into the flash ;) )!</p></div>
                            </div>
                        </li>
                    
                        <li>
                            <h2><a href="/project/6855/log/21636-update-4-code-overhaul-github-repo-coming-soon">update 4: code overhaul! + github repo coming soon</a></h2>
                            <small class="authorship">21 days ago  &bull;
                                <a class="gray-link" href="/project/6855/log/21636#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-21636"><p>the whole system is being rewritten as of now, and I&apos;m using more classes, as this keeps code nicer to look at, but also increases security on the user app-side:</p><p><br><br>it was this before:<br>all system variables and functions can be adjusted/called from your own apps. to prevent this from happening, simply declare all variables in the &apos;private&apos; section of your app.<br>And for the functions, recommended is using __ in front of every function you make, example: &apos;void __init_vars()&apos;.</p><p><br><br>and now it is:<br>no system variables can be accessed, and only functions that are set as &apos;public&apos; can be accessed from apps now :D<br>this enhances security of the watch.</p><p><br><br>this also means one can not access the graphics library directly (including all driver chip commands), so a &apos;wrapper&apos; is being written as well.</p><p>the gihub repo will hopefully be online by the end of this or next week<br></p></div>
                            </div>

                        <li>
                            <h2><a href="/project/6855/log/21993-update-7-more-battery-optimizations">update 7: more (battery) optimizations</a></h2>
                            <small class="authorship">16 days ago  &bull;
                                <a class="gray-link" href="/project/6855/log/21993#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-21993"><p>optimizing battery span can be quite difficult, but not impossible<br>even when the screen is &apos;fully&apos; off, the boost converter to 13V for the screen power still runs, making the total amount of consumed current about 100mA (when screen is fully black)<br>so the only solution is to include a hardware switch which can power the screen fully down (sadly this also affects the microSD slot).</p><p><br><br>same concept is going to be applied to the bluetooth to save more battery. :)</p><p><br><br>so basically: Atomsoft and i are getting pretty close to a final schematic for the whole thing.</p><p><br><br>with current software optimizations, the 1400mah lipo battery can now last up to 24 hours till it reaches 3,3V (3,7V is recommended lowest level, so 3,3V is danger)</p><p><br><br>and more functionality will be added very soon to the code, allowing for more graphic commands, accelerometer readings and such!</p><p><br><br>also, the website home will undergo some slight changes, with links to this hackaday project and such :)</p><p>and if that isn&apos;t enough for this update: the 3D case design is getting along pretty neatly, the dimensions are 50*31*9mm as of now. (including strap connectors)</p><p>and without strap connectors: 42*41*9mm</p></div>
