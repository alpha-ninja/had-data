                            <small class="authorship">2 months ago  &bull;

                            <div style="position:relative">
                                <div id="post-body-19156"><p>I think it is time to share my development progress with everyone, I have updated the project with the GIT repo link from Bitbucket.</p><p>What is included :</p><ol><li>aleph-lan-embedded<br>This project is developed using Eclipse-CDT and ARM GCC GNU. I am doing the development under ubuntu. The project contains all the source code and binaries for the Texas Instruments connected launchpad. <br><br>Everything is working on this side, what remain is to link the functionalities together. Receive the POST method on LAN and send it to NFC. This will be done once the PC side is completed.<br><br></li><li>aleph-lan-pc-config<br>This tool will be responsible for installing and configuring everything on the PC. For now it is able to automatically identify Aleph on the network. The rest is ready as cmd commands, will just need to implement it in Java. <br><br></li><li>aleph-lan-pc-print<br>This is the main tool of Aleph. This tool will be running on the PC, it will catch the print command and parse it according to predefined steps in the Java Code. It will then send the parsed data as JSON to the Aleph device and upload a PDF copy to the server.<br><br></li><li>aleph-lan-hardware<br>For now this folder only contains the reference design of the development boards. Soon it will contain my own PCB design using KiCad.<br><br></li><li>aleph-lan-mobile<br>Not yet created, in the future it will contain the mobile app code for Android. The app will receive the NFC from Aleph and provide it to the user<br><br></li></ol><p>The code is not yet cleaned or commented, this is just the start. I am not a Java programmer so my code might not be the most optimized but it will do the trick for now.</p></div>

                            <h2><a href="/project/4665/log/20007-hardware-description">Hardware Description</a></h2>
                            <small class="authorship">2 months ago  &bull;
                                <a class="gray-link" href="/project/4665/log/20007#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-20007"><p>All the testing have been done with the <a data-mention-user-id="56439" href="https://hackaday.io/hacker/56439">@Texas Instruments</a> development boards and the Alpha version is up and running. This means that it is time to move the design to the Beta version. </p><p>In the Alpha phase the device was based on:</p><p>1- Connected Launchpad <a href="http://www.ti.com/ww/en/launchpad/launchpads-connected-ek-tm4c1294xl.html#tabs" target="_blank">EK-TM4C1294XL</a><span class="redactor-invisible-space"><a href="http://www.ti.com/ww/en/launchpad/launchpads-connected-ek-tm4c1294xl.html#tabs"></a></span></p><p>2- NFC Evaluation Module <a href="http://www.ti.com/tool/trf7970atb" target="_blank">TRF7970ATB</a><span class="redactor-invisible-space"><a href="http://www.ti.com/tool/trf7970atb"></a></span></p><p>The TM4C role is connectivity. It is able to connect the device to the LAN with DHCP support. After the connection has been established it then waits for a POST request from the printing server. </p><p>Once the POST request has been received it then parses the request and passes it to the NFC module TRF7970A.</p><p>The NFC chip role is to simply transfer the received data to the mobile phone. One tap and your parsed receipt is on your mobile for additional calculations and so on.</p><p>The device is powered from a micro USB port so the input voltage to the board is at +5V. The TM4C runs at +3.3V so for sure we need a DC/DC converter. To keep the noise as its minimal levels, I have chosen a simple LDO to do the task especially that the voltage drop is minimal. For this the <a href="https://store.ti.com/TPS73733DRVT-Single-Output-LDO-1A-Adj-12-to-50V-Reverse-Current-Protection-P5483.aspx" target="_blank">TPS73733DRVT</a><span class="redactor-invisible-space"> will do the trick.</span></p><p>As a summary here is how the block diagram looks like:</p><p><img src="https://cdn.hackaday.io/images/9327731435309712063.png"></p><p>Based on the design model above, the schematic design for the Beta version was finished and reviewed. You can check the full design in the GIT repository of this project. There you will find the KiCad project, some libraries and dxf drawings. If you don't have KiCad installed there is a PDF file of the schematic included or you can also check the images below.</p><p><img src="https://cdn.hackaday.io/images/2018271435310082000.jpg"></p><p><img src="https://cdn.hackaday.io/images/8042391435310120124.jpg"></p><p><img src="https://cdn.hackaday.io/images/6581771435310153153.jpg"></p><p>The components details will be updated in the BOM list in the project page once it is ready.</p><p>Special thanks to <a data-mention-user-id="57212" href="/hacker/57212">@JosephAbouElOula</a> for the schematic review</p></div>

                            <h2><a href="/project/4665/log/20334-reply-to-the-meetup-feedback">Reply to the meetup feedback</a></h2>
                            <small class="authorship">2 months ago  &bull;
                                <a class="gray-link" href="/project/4665/log/20334#discussion-list">
                                    0 comments
                                </a>
                                </small>
                            <div style="position:relative">
                                <div id="post-body-20334"><p>I wasn't able to attend the meetup due to timezone difference. But I read all the feedback and here are my comments :</p><p style="margin-left: 40px;"><a data-mention-user-id="2501" href="/hacker/2501">@xobmo</a> says :<br>Aleph, honestly should be a smartphone app, nothing more, but that's not hardware, and it's not my project<br><a data-mention-user-id="2501" href="/hacker/2501">@xobmo</a> says:<br>square works but is annoying, nfc is in every new smart phone, tap to pay, tap for receipt<span class="redactor-invisible-space"><br><br>Reply:<br>It will be great to be able to tap to pay and tap for receipt. But saying that is something and actually implementing it is something else.<br>Without hardware modifications on the 'tap to pay' machine this is not feasible, and that is what Aleph is. It doesn't have to be a device on its own, it is the hardware responsible for the tap to receipt part. In the future it can be integrated within a tap2pay device without a problem.<br><br>Aleph cannot be a smartphone application only. You need a PC side software along with the hardware part that I discussed above. Without the PC side software, how would the POS tell the 'tap2pay' NFC machine about the receipt details ? The PC side software grabs the print command parses it and sends it to the Aleph hardware. Without it, you will need to modify every existing POS software to add the feature. I think this will be a big limitation. </span></p><p style="margin-left: 40px;">The last point is that from the customer POV, Aleph is only a simple mobile/web application.</p><p style="margin-left: 40px;"><a data-mention-user-id="4951" href="/hacker/4951">@j0z0r pwn4tr0n</a> says:</p><p style="margin-left: 40px;">I think if it was low cost enough (like &lt;$100) it would offset the cost of the thermal paper and stores would be interested. I know consumers would be interested because then they could export their expenses to mint or take care of business expenses with a click instead pulling out a wad and hitting the scanner<br></p><p style="margin-left: 40px;">Reply: <br>The hardware cost at this point is very minimal, around 20$ as a top margin for it. Adding the remaining costs and profits and many other things I am not an expert on, I think it still should be a competitor to thermal printers cost-wise.</p><p style="margin-left: 40px;"><a data-mention-user-id="16981" href="/hacker/16981">@Blecky</a> says :<br>As long as it's streamlined enough and the customer has control over it it should be fine. For example, if you don't tap your phone within a few seconds, it prints the receipt instead<span class="redactor-invisible-space">.<br></span></p><p style="margin-left: 40px;"><span class="redactor-invisible-space"><br></span></p><p style="margin-left: 40px;"><span class="redactor-invisible-space">Reply:<br></span>This is a great idea, I think that Aleph should be very flexible. And implementing new features such as this one should be straight forward in the final design<span class="redactor-invisible-space">. I think I should leave a margin for the store owner to decide how he would want to interact with Aleph.</span></p><p style="margin-left: 40px;"><span class="redactor-invisible-space"><span class="redactor-invisible-space"></span></span><a data-mention-user-id="28790" href="/hacker/28790">@LloydTCannonIII</a> says:</p><p style="margin-left: 40px;">I love the Idea Aleph, would help disorganized budgeters so much, if I had a smart phone I would definitely use it</p><p style="margin-left: 40px;"><a data-mention-user-id="4951" href="/hacker/4951">@j0z0r pwn4tr0n</a> says:<br>Maybe for the last of the Aleph feedback: Make it so people without a smartphone can use it, like just a tag or something that e-mails it back home<br></p><p style="margin-left: 40px;">Reply:<br>Never thought about none smartphone users before.
